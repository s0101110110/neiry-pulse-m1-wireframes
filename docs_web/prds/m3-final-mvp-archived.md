# PRD · M3 — Final MVP (Neiry Pulse)

**Версия:** v1.0
**Дата:** 2026-06-05
**Автор:** Костя (PM)
**Дедлайн релиза:** 2026-06-25 (20 дней от старта)
**Источник видения:** `knowledge-base/INBOX/external/nikita-vision.processed.html` (Никита)
**Источник пост-мортема M2:** `docs_web/reports/post-village-2026-05.html`

## 1. Цель M3

Сделать **полнофункциональный MVP** Neiry Pulse, готовый к демо для каждой целевой ниши (Sport-лига и Corporate-клиенты). После M3 продукт можно показывать платящим клиентам, а не только инвесторам.

**Бизнес-ставка:** M2 показал техническую жизнеспособность на стенде. M3 должен закрыть цикл «купил → подключил → получаю ценность ежедневно» хотя бы для двух сегментов.

**Уточнение:** «Платящие клиенты» = за деньги, не бесплатные пилоты.

## 2. Скоуп M3

### 2.1 Mobile App — 2 Mode

Mode выбирается при регистрации, зашит в учётной записи.

| Mode | Кто | Что внутри |
|------|-----|-----------|
| **Sport** | Спортсмен лиги (бег / триатлон / вело / лыжи / плавание / кроссфит) | Тренировки, GPS с телефона, пульсовые зоны, готовность к нагрузке, лига-рейтинг |
| **Corporate** | Сотрудник компании — office + safety-роли | Стресс / Recovery / Burnout · для safety-ролей дополнительно: детектор сонливости, алерты |

### 2.2 Backend

- Mode enum в учётке
- Расширение биометрики: **добавление сна** (root cause из M2 закрыт — см. R&D-карточку 3)
- Sport workers (Celery): тренировки, пульсовые зоны, GPS-обработка
- Safety workers: детектор сонливости (на raw IMU + ЧСС), алерт-движок
- Push-уведомления для алертов сонливости и тренировочных напоминаний (FCM Android + APNS iOS, обе платформы в W4)
- Алерт-правила safety — конфиг на клиента (порог сонливости, время реакции, каналы доставки)
- Privacy policy / Terms — пишет юрист (предоставляет Neiry)
- **iOS Bridge для китайского SDK** (Кирилл, W2): нативный мост, чтобы то же ядро mobile-app работало на iPhone (на M2 отказались — теперь делаем)
- **iOS-билд для App Store** (W3-W4): второй билд того же приложения под Apple — одна кодовая база, два билда (Google Play + App Store)
- **Подготовка и публикация билдов в App Store и Google Play** (PM, W3-W4): аккаунты разработчиков, метаданные, скриншоты, политика конфиденциальности, прохождение review — часть acceptance M3

### 2.3 Дашборды — 2 пресета (НЕ 3 таба!)

| Пресет | Домен | Сегмент | Внутреннее устройство |
|--------|-------|---------|----------------------|
| **Лига** | `league.neiry.ru` | Спорт-сообщества, беговые клубы, триатлон, лыжи | Рейтинг (League Fitness Score), список участников, графики тренировок, личные рекорды, отдельный брендинг |
| **Корпорат** | `dashboard.neiry.ru` | Корпораты — office + safety одновременно | Универсальное ядро + role-based фильтр (office → стресс/burnout · safety → сонливость/алерты). НЕ два отдельных таба |

**Общее ядро дашбордов:** auth, список юзеров, графики, экспорт. Кастомизация — через пресет (брендинг, метрики на главной).

**White-label lite (для обоих пресетов):** оба пресета должны поддерживать брендирование под B2B-клиента — лого, акцент-цвет, кастомный домен. Конфигурацией, без релиза. Heavy white-label (кастомные метрики/настройки) — out-of-scope M3.

**League Fitness Score:** формула выносится в отдельную R&D-карточку (owner — R&D, дедлайн W2).

### 2.4 Демо-готовность

- Демо-аккаунт под каждый пресет: «спортсмен Sport-Mode» + «офисный сотрудник» + «водитель safety»
- Демо-данные = живые, с 3 реальных noser'ов (start ношения не позже W3 — минимум 7 дней до съёмки роликов)
- Демо-ролики и питч-материалы под каждый сегмент: deck + one-pager + видео
- Готовность парка браслетов к показу клиентам (количество TBD — зависит от подтверждения пилотов Никитой)

**Параллельно (PM):**

- Privacy / consent bench (Whoop, Oura, Polar) → требования к onboarding mobile (R&D-карточка 5)

### W2 · 2-15 июня

**Sport Mode + Лига:**

- Дашборд Лиги на `league.neiry.ru`: финальный дизайн, спецификация League Fitness Score, демо-данные (мок-сезон)

### W3 · 9-22 июня

**Corporate + Safety + Sleep:**
- Corporate Mode на mobile: ввод роли (office / водитель / машинист / диспетчер) при создании аккаунта HR
- Safety-роли: детектор сонливости, алерт-флоу, push-уведомления
- Дашборд Корпорат: интеграция role-based фильтра, sleep-блок для office
- Передача браслетов в корп-пилот, регламент ношения

### W4 · 16-25 июня

**Push, шлифовка, релиз M3:**
- Push-уведомления (alerts safety + reminders sport)
- Финальное QA на 2 пресетах
- Демо-аккаунты + ролики + питч-материалы под каждый сегмент
- Готовность к публичному показу клиентам

## 4. Acceptance Criteria

M3 считается достигнутой, когда:
- [ ] Mobile-app позволяет зарегистрироваться в одном из 2 Mode (Sport / Corporate)
- [ ] Sport-юзер видит на mobile свои тренировки, GPS-трек, пульсовые зоны
- [ ] Corporate-юзер видит свой стресс, recovery, burnout; safety-роль дополнительно видит алерты сонливости
- [ ] Сон отдаёт корректные значения Deep / REM / Light (исправление M2-фейла)
- [ ] `league.neiry.ru` поднят, показывает рейтинг + список + графики хотя бы для одного пилота
- [ ] `dashboard.neiry.ru` показывает корп-команду с role-фильтром (office/safety)
- [ ] Push-алерт сонливости приходит на телефон safety-юзера при триггере
- [ ] Под каждый пресет есть демо-аккаунт + 2-3 ролика
- [ ] Минимум 1 пилот в каждом сегменте получил браслеты и активно использует (минимум 3 дня ношения за W4)
- [ ] Mobile-app опубликован в Google Play и App Store (или прошёл review и доступен для распространения)
- [ ] Оба пресета (Лига и Корпорат) брендируются под клиента: лого, акцент-цвет, домен — конфигурацией без релиза (white-label lite)

## 5. Out-of-scope (НЕ делаем в M3)

- Health-линейка устройств (VANTA, ECG-браслет) — приходит позже июня
- Социалка в Sport (friends, чаты, лайки) — post-M3
- Анализатор калорий, женский цикл, служба заботы (V2-фичи, post-M3)
- 3-табная архитектура корп-дашборда (отвергнута — следуем Никите: 2 пресета)
- Multi-role у одного сотрудника (диспетчер + office одновременно) — post-M3
- Двойной аккаунт (один человек в Sport и Corporate одновременно) — не B2B-сценарий
- White-label heavy (кастомные метрики, настройки клиента) — post-M3
- B2C-каналы продаж — M3 остаётся B2B
- tVNS / Recovery devices (Headband, tVNS) — явно не в M3

## 6. Зависимости и риски

### Зависимости
- **Сон:** root cause M2-фейла → выбор «SDK-fix vs свой actigraphy-алгоритм» к W1 (см. R&D-карточку 3)
- **Vigor IMU/PPG raw:** для детектора сонливости и actigraphy-сна — sample от вендора нужен к W2 (см. R&D-карточку 2)
- **ICP v2:** влияет на коммуникацию M3-питча, не блокирует разработку
- **Privacy:** влияет на дизайн mobile onboarding (W2-W3), требует bench-анализ от R&D
- **Аккаунты пилотов:** Никита подтверждает спорт-лигу и корп-клиента (W2-W3)

### Риски
| Риск | Митигация |
|------|-----------|
| Сон так и не починим к W3 | Резерв: показываем только grade «Достаточный/Недостаточный» из IMU-actigraphy, без Deep/REM/Light |
| Vigor не отдаст raw IMU/PPG вовремя | Резерв: оставляем существующую int8-логику, сонливость считаем на BPM-derivatives |
| Push на Android не успевает к W4 | Митигация: в первой итерации M3 — только in-app алерты; push на пост-M3 |
| Лига-пилот не выходит на связь | Резерв: используем демо-сезон с мок-данными, реального пилота переносим на post-M3 |
| **R-104:** Перегрузка Кирилла (iOS+Android параллельно W2-W4) | К W2 — оценка с Кириллом «тяну/не тяну». Если нет — найм fullstack контрактника на iOS-стрим (бюджет TBD). Без решения к W2 — iOS-build уходит на post-M3 |
| **R-105:** App Store review >7 дней | Подаём билд не позже 18.06 (на 7 дней раньше дедлайна). Если review >5 дней — №10 acceptance переходит в статус «submitted, awaiting review» (не блокирует релиз M3 если всё остальное ✅) |
| **R-106:** Юрист не успевает к privacy policy | W1: Neiry предоставляет юриста. Fallback: template от Termly/Iubenda, адаптируем под Neiry. Худший случай — публикуем через шаблон без custom-policy |
| **R-107:** White-label lite не успеваем в W4 | Если в W4 не успеваем — №11 acceptance понижаем до «1 пресет (Лига или Корпорат) брендирован», второй — post-M3 (1 неделя работы) |
| **R-108:** 3 живых noser'а не выбраны к W3 | W2: PM назначает 3 человек (вариант: ты + 2 из команды). Если не найдены — fallback: синтетические данные + дисклеймер на ролике |
