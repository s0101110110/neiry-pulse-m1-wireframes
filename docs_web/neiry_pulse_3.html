01Три milestone

M1 — техническое подтверждение пайплайна. M2 — публичная выставка с минимальной функциональностью. M3 — полнофункциональный MVP.

M1 · 18 мая
Tech Demo
~11 дней · без дизайна
Проверить, что end-to-end пайплайн жив. Браслет → мобайл → backend → дэшборд. Без Mode, без графиков. Параллельно три потока работ.
  • Mobile (Android + iOS): регистрация + BLE pairing + sync пульса
  • Backend: FastAPI + PostgreSQL, базовые endpoints
  • Dashboard: логин админа + список юзеров
  • Yandex Cloud VM, NGINX, certbot, Sentry
  • 3–5 реальных VITRO браслетов передают реальные данные
  • Синхронизация со смежными командами Mindtracker и Stim-X
M2 · 25 мая
Startup Village
~7 дней после M1 · с дизайном
Публичная выставка. На дэшборде и в приложении должны быть графики и минимальная функциональность. Wow-эффект для инвесторов и партнёров.
  • Графики биометрики на дэшборде (live + история)
  • Базовый main-экран в приложении с графиком пульса
  • Минимально красивый UI (не голый Material)
  • Демо-сценарий: «вот спортсмен, его данные, рейтинг»
  • Расширение биометрики: ВСР, шаги
  • Демо-стенд: live-демонстрация на посетителях
M3 · 25 июня
Финальный MVP
~30 дней после M2
Работающий продукт с 2 Mode (Corporate, Sport) и 2 пресетами дэшборда (Лига, Корпорат). Готовность к демо для каждой ниши.
  • Mobile: 2 Mode (Corporate, Sport)
  • Universal Dashboard с 2 пресетами
  • Биометрика: пульс, ВСР, шаги, сон (с условной обработкой)
  • Тренировки, GPS с телефона, готовность к нагрузке
  • Метрики безопасности: сонливость, утомление, готовность к смене
  • Push-уведомления (FCM + APNS)

04Tech Demo · 2 недели

Цель — техническое подтверждение пайплайна. Главный артефакт: видео или live-показ — «человек надел браслет, открыл приложение, через 5 минут его пульс отображается на дэшборде у админа».

Что должно работать

  • Mobile (Flutter): регистрация по email/пароль, BLE pairing с VITRO, синхронизация пульса в backend каждые N минут
  • Backend (FastAPI): приём данных, сохранение в PostgreSQL, базовые endpoints (auth, sync, list)
  • Web Dashboard: логин админа + одна страница со списком юзеров (имя, последняя синхронизация)
  • Infrastructure: Yandex Cloud VM, NGINX + certbot, GitHub Actions, Sentry
  • 3–5 реальных VITRO браслетов на тестировщиках, передающих реальные данные

Что НЕ нужно в Tech Demo

  • Дизайн — стандартные компоненты Flutter и фронта
  • Mode-переключение, выбор роли, разные виды юзеров
  • Графики, тренировки, GPS, протоколы, alerts
  • Дополнительная биометрика (ВСР, сон, шаги)
  • Многотенантность, ограничение видимости по группам
  • Push-уведомления

06Синхронизация со смежными командами

Mindtracker (Headband) и Stim-X (tVNS) — продукты той же организации. У них уже есть наработанная инфраструктура и компоненты, которые можно переиспользовать. Это критично для срока.

Задача · Неделя 1 · Owner: Константин

Цель: понять, что переиспользуем, что строим с нуля

Провести встречи с командами Mindtracker и Stim-X. Цель — выяснить, какие компоненты у них уже работают, и согласовать, что можно использовать в Neiry Pulse. Результат — документ с inventory и согласованным планом интеграции.

Что выясняем на встречах:

Регистрация и auth Есть ли единый Neiry Account? Можно ли подключиться через SSO/OAuth?
Профиль пользователя Какие поля общие? Где хранится (БД, схема)? Можно ли переиспользовать?
API контракты телеметрии Есть ли единый стандарт передачи биометрики? Можем ли согласовать общую схему?
Push-сервис Один FCM/APNS аккаунт на всю экосистему? Кто управляет ключами?
Дизайн-система Есть ли общий design tokens, компоненты, гайдлайны? Или строим свой?
Биллинг и подписки Общая модель монетизации? Единая платёжная система?

Стратегия: то, что у Mindtracker / Stim-X уже работает в проде — берём как есть или с минимальной адаптацией. То, чего нет — строим в Neiry Pulse, но в архитектурно совместимом виде, чтобы потом не мигрировать данные.

Артефакт: документ Neiry_Ecosystem_Sync.md с тремя секциями: «Что переиспользуем», «Что строим с нуля», «Точки совместимости на будущее».

07HBandSDK Inventory

Оригинальное китайское приложение G BAND детально проработано на той же SDK. Чтобы не пилить заново то, что уже работает в SDK, в первую неделю нужно сделать аудит.

Задача · Неделя 1 · Owner: Кирилл

HBandSDK Inventory Document

Параллельно с настройкой инфраструктуры составить документ с полным инвентарём возможностей SDK.

Что должно быть в документе:

  • Все методы SDK с пометкой: даёт готовые данные или сырые сигналы
  • Какие типы тренировок/активностей поддерживает SDK нативно
  • Какие расчёты SDK делает на устройстве: ВСР, VO2max, калории, фазы сна
  • Скриншоты всех экранов G BAND с описанием UX-логики
  • Рекомендация: что переиспользуем, что пилим сами

Зачем: если SDK считает фазы сна или поддерживает workout type «бег» — не пилим. Это экономит недели разработки.

10Пресеты дэшборда · 2 штуки

Универсальный дэшборд с общим ядром (auth, список юзеров, графики, экспорт) и 2 пресетами. Лига — на отдельном поддомене с брендом. Корпорат — на dashboard.neiry.ru, включает офис и safety-роли.

  • Недельный километраж (Weekly Mileage)
  • Распределение пульсовых зон (HR Zones)
  • Лидерборды лиги

  • Тренд стресса (Stress Trend)
  • Командные агрегаты с анонимизацией
  • Детектор сонливости (Drowsiness Detection)
  • Индекс утомления (Fatigue Index)
  • Готовность к смене (Pre-shift Readiness)

Как работает «Корпорат» одним пресетом

Один дэшборд, два сценария использования. По умолчанию (роль «Офис») — фокус на выгорании, стрессе, командных агрегатах. Если у юзеров выбрана safety-роль (водитель, машинист, диспетчер) — на overview добавляются виджеты сонливости, утомления и готовности к смене. HR / специалист по охране труда видит то и другое в одном интерфейсе, фильтрует по роли.

Архитектура «гибрид поддоменов»

league.neiry.ru — отдельный поддомен с брендом лиги. Это нужно для продаж: лига получает «свой» продукт, можно дать white-label под конкретного клиента в Волне 2.

dashboard.neiry.ru — единая точка входа для корп-сегмента. Технически — та же React-сборка, тот же codebase. Разница — env-конфиг и роутинг NGINX.

12Онбординг

Самый простой возможный флоу. Mode определяется выбором роли при регистрации.

Шаги первого запуска приложения

  1. Splash + onboarding скрины (2–3 экрана о Neiry Pulse)
  2. Регистрация по email или телефону + пароль
  3. Экран выбора роли (всего 2 варианта в MVP):
    • — «Я сотрудник компании» → Corporate Mode
    • — «Я спортсмен» → Sport Mode
  4. Базовый профиль: имя, год рождения, пол, рост, вес
  5. Pairing браслета по BLE (модель определяется автоматически)
  6. Mode-специфичные шаги:
    • — Sport: максимальный пульс, основной вид спорта (бег / триатлон / вело / лыжи / другое), целевое соревнование
    • — Corporate: код компании (опционально); ФИО для HR; роль (офис / водитель / машинист / диспетчер) выбирает HR при добавлении

16Риски и митигации

Риск
Не уложимся в 8 недель MVP
Митигация: первым на отказ — push-уведомления и safety-метрики. Sport + офисный корп — must-have.
Риск
iOS BLE-bridge сложнее Android
Митигация: параллельная разработка в Tech Demo. Закроем риск на неделе 2.
Риск
Детектор сонливости требует валидации
Митигация: в MVP — простая модель (ВСР + микро-паттерны). Полная валидация — Волна 2.
Риск
Sport включает много видов спорта
Митигация: в MVP — типизация workout, без специфичных метрик для каждого вида.
Риск
APNS — задержка одобрения от Apple
Митигация: настроить APNS параллельно с iOS-bridge на Tech Demo неделе.
Риск
Sync со смежными командами затягивается
Митигация: жёсткий дедлайн неделя 1. Если не успели — строим своё, согласуем интеграцию в Волне 2.
