# Anti-Generic Design Pipeline — Research & Pipeline Design

**Дата:** 2026-07-01
**Автор:** PM-агент (дирижёр) + PM (заказчик)
**Статус:** Design approved → переход к writing-plans
**Режим исполнения:** ultracode (многоагентный Workflow)

---

## 1. Контекст и проблема

На звонке команды (Никита, Дымов, другие PM) дизайн Neiry Pulse (bevel-clone) получил
предметную критику. Четыре симптома:

1. **«Типовые компоненты»** — UI-элементы выглядят как стоковые shadcn-дефолты без характера.
2. **«Нарушена композиция»** — сетка, ритм, иерархия, плотность.
3. **«Поганый wine-цвет»** — претензия к применению wine `#831843`.
4. **«Не выглядит как рукотворный»** — общее ощущение AI-генерата, не работы дизайнера.

PM (заказчик этой работы) хочет опровергнуть оценку **не обороной, а реальным
level-up** — выстроив пайплайн, который системно производит «рукотворный»,
не-generic результат, и доказать это flagship proof-экраном.

### Ключевая находка контекста
- Скиллов `taste` (tasteskill.dev) и `UX UI Pro Max` (ui-ux-pro-max-skill.com)
  **у нас НЕТ** — это внешние продукты по ссылкам.
- Зато установлен мощный арсенал в `UI_assets/.agents/skills/`:
  `high-end-visual-design`, `emil-design-eng`, `minimalist-ui`,
  `redesign-existing-projects`, `imagegen-frontend-mobile`,
  `app-store-screenshots` + `impeccable` (в `.claude/skills/`).
- Тезис PM «у нас все крутые скиллы, нужна правильная имплементация» —
  буквально верен. Проблема — в **оркестрации**, не в наличии инструментов.
- Текущий пайплайн (PM пишет BRIEF → UX-агент верстает HTML/Tailwind →
  proof PNG → acceptance → commit) **системно не задействует эти скиллы** —
  по lessons-learned UX-агента скиллы в процессе не фигурируют.

## 2. Цели и критерии успеха

### Цели
- **G1.** Research по `taste` и `UX UI Pro Max`: методология (что делает их сильными)
  **+ verdict** «брать/не брать/как доступны».
- **G2.** Через Parallel CLI найти ещё более сильные скиллы и методологию
  «как оркестрировать design-скиллы против generic-AI-look».
- **G3.** Собрать оптимальный **pipeline** из наших скиллов + находок,
  заземлённый на 4 пункта критики, с распределением ролей PM / UX-агент / дирижёр.
- **G4 (отложено).** Flagship proof-экран bevel-clone по этому пайплайну —
  отдельной фазой после плана.

### Критерии успеха
- ✅ Каждый из 4 пунктов критики имеет конкретную stage-контрмеру в пайплайне.
- ✅ Есть verdict «брать/не брать» по 3 внешним инструментам (taste, UX UI Pro Max, Stitch)
  с обоснованием, ценой и доступностью.
- ✅ Пайплайн исполним UX-агентом — указаны точные скиллы и аргументы по стадиям.
- ✅ Proof-экран (фаза G4) проходит anti-generic gate и держит side-by-side
  с реальным Bevel.

### Мандат (решение PM)
**Честный level-up — canon можно пересматривать.** Если research/диагностика
покажут, что wine `#831843`, типографика (Golos Text) или bevel-clone v3 палитра
частично виноваты — предлагаем альтернативы. Canon не священен; результат > эго.
Затрагивает зафиксированные в памяти «незыблемые» решения — каждое предложение
по их пересмотру оформляется явно, с обоснованием, под решение PM.

## 3. Дизайн исследования (fan-out)

Подход 🅐 — **diagnostic-first**: диагностика заземляет research на реальную боль,
чтобы не выдать «generic best-practices» (ровно то, за что критикуют).

### Поток 0 · Диагностика (запускается ПЕРВЫМ, питает остальные)
Teardown текущего `bevel-clone/index.html` + home-экрана, сверка с 14 реальными
скринами Bevel (`UI_assets/screenshots/references/`). Выход — таблица failure modes,
привязанная к критике:

| Критика | Что ищем |
|---|---|
| типовые компоненты | стоковые shadcn-карточки/кнопки/радиусы без характера vs кастом Bevel |
| нарушена композиция | сетка, ритм, иерархия, плотность, выравнивание |
| поганый wine | площадь / контраст / where & why применения; сравнение с палитрой Bevel |
| не рукотворный | микро-типографика, оптические отступы, «несовершенства», тени, бордеры |

### Параллельные потоки research (после диагностики)

- **A · Внешние скиллы** — `tasteskill.dev` + `ui-ux-pro-max-skill.com`:
  что это, методология, цена, доступность → методология + verdict.
- **B · Наш арсенал** — чтение 7 SKILL.md → матрица «скилл × какой симптом лечит».
- **C · Broad sweep (Parallel CLI)** — «ещё более крутые design-скиллы» +
  методология «оркестрация скиллов против generic-AI-look».
- **D · Bevel DS** — поиск официального UI-кита / дизайн-системы Bevel в вебе +
  анализ 14 референсов: что делает Bevel «рукотворным».
- **E · Референсы** — разбор `screenshots/references` как эталона: переиспользуемые
  паттерны/токены, которые наш клон растерял.
- **F · Stitch × Figma MCP** — `stitch.withgoogle.com`: прикладной сценарий в нашем
  пайплайне. Связка `Stitch (генерация композ-вариантов) → Figma (MCP уже работает) →
  код`. Verdict «усиливает / мешает». NB: Figma MCP в проекте подключён и работает.

Все потоки → синтез → пайплайн (раздел 4).

## 4. Пайплайн — Anti-Generic Design Pipeline

Скелет (research наполнит конкретикой). Принцип: каждая стадия закрывает
конкретный симптом, имеет владельца-роль и привязанный скилл.

| # | Стадия | Владелец | Скилл(ы) | Лечит симптом |
|---|---|---|---|---|
| 1 | **Intent & Taste-target** — бриф + выбор эталона из Bevel-рефов/DS ДО пикселей | PM + дирижёр | `taste` принципы | задаёт планку, гасит generic на входе |
| 2 | **Composition-first** — сетка/ритм/иерархия раньше компонентов; опц. Stitch → Figma MCP → отбор | UX-агент | `high-end-visual-design`, Stitch/Figma | нарушена композиция |
| 3 | **Bespoke components** — замена стоковых shadcn-дефолтов на кастом (радиусы, бордеры, био-виджеты) | UX-агент | `emil-design-eng`, `minimalist-ui` | типовые компоненты |
| 4 | **Color & hand-crafted detail** — пересборка wine (площадь/контраст/роль), оптические отступы, тени, «несовершенства», mono-числа | UX-агент | находки `taste`/`UX UI Pro Max` | поганый wine + не рукотворный |
| 5 | **Anti-generic gate** — adversarial-критика «читается ли как AI-генерат? отгрузил бы дизайнер уровня Bevel?», скоринг против рефов, петля назад при фейле | UX-агент / критик | `impeccable critique`, `taste` | ловит generic до глаз PM |
| 6 | **Proof & acceptance** — proof PNG side-by-side с Bevel-рефом, PM acceptance gate | дирижёр + PM | — | acceptance |

### Роли
- **PM (заказчик):** intent, выбор taste-target, финальный acceptance.
- **UX-агент:** генерация, компоненты, цвет, детали, прогон критик-скиллов.
- **Дирижёр (PM-агент):** оркестрация, синтез диагностики/research, брифы,
  enforcement anti-generic gate, сборка proof.

## 6. Модель исполнения

- **Режим:** ultracode → исполнение research-фазы как многоагентного **Workflow**:
  диагностика (Поток 0) → barrier → параллельные потоки A–F → adversarial-верификация
  находок → синтез.
- **Делегирование:**
  - Web-потоки (A, C, D, F) — purpose-spawned параллельные агенты (general-purpose /
    Explore) либо скилл `deep-research`.
  - Визуальная диагностика (Поток 0) + разбор референсов (E) — через **UX/UI-агента**
    (PM-role-boundary: визуальную оценку не делает дирижёр сам) или `impeccable critique`.
  - Аудит арсенала (B) — чтение SKILL.md, делает дирижёр инлайн.
- **НЕ использовать** научный R&D-агент (`NEIRY/research/`) — он под BCI/нейрофизиологию,
  это domain mismatch.
