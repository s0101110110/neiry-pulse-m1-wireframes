# Дизайн: Мульти-агентная PM-система Neiry

**Дата:** 2026-05-16  
**Автор:** Константин Соломонов  
**Статус:** Утверждён

---

## Контекст и проблема

Проект Neiry Pulse ведётся в нескольких папках (`docs_web`, `dashboard`, `UI_assets`, `backend`, `mobile`). Материалы накоплены в разных форматах: PRDs, ТЗ, транскрипции встреч, roadmap, Excel-таблицы, Google Docs. Три ключевые боли:

1. **Потеря контекста** — каждую сессию приходится объяснять агенту что происходит заново
2. **Разрозненность материалов** — непонятно где искать принятые решения
3. **Выпадение задач** — договорённости остаются в транскрипциях, не попадают в трекер

---

## Решение: Гибридная мульти-агентная система

Пять агентов с изолированными ролями + единый центр знаний на базе Obsidian. PM-оркестратор может делегировать задачи субагентам. Всё хранится в markdown-файлах — токен-эффективно, версионируется через git.

---

## Секция 1. Агенты и роли

| Агент | Папка | Роль |
|---|---|---|
| 🎼 PM / Оркестратор | `docs_web/` | Управляет проектом, памятью, задачами. Делегирует субагентам. Обновляет инструкции других агентов |
| 🎨 UX/UI Дизайнер | `UI_assets/` | Создаёт wireframes и макеты на основе UI-ассетов и UI-китов |
| 💻 Frontend Developer | `dashboard/` | Реализует код по брифам от PM |
| 🔬 R&D Researcher | `research/` | Накапливает знания по железу, SDK, метрикам здоровья. Объясняет сложное простым языком |
| 📚 Knowledge Base | `knowledge-base/` | Не агент — Obsidian vault, общий центр знаний для всех |
| 🧠 Second Brain | `second-brain/` (отдельный репо) | Личная база знаний (wiki, marketing). Отдельный Obsidian vault. Neiry KB может ссылаться на него, но не сливается |

**Границы:**
- `mobile/` и `backend/` — зона Кирилла, агенты не трогают
- Только PM имеет право обновлять `CLAUDE.md` других агентов
- R&D Researcher и PM пишут в knowledge-base; остальные только читают

## Секция 2. Трёхуровневая память

### Уровень 1 — Всегда в контексте (~50 строк, минимум токенов)
`CLAUDE.md` каждого агента: роль, принципы, путь к `PROJECT_STATE.md`. Не растёт.

### Уровень 2 — Читается в начале каждой сессии (~100-150 строк)
`knowledge-base/PROJECT_STATE.md` — живой снимок:
- Текущий milestone и дедлайн
- Открытые задачи по участникам
- Последние 3-5 решений
- Нерешённые вопросы

Обновляется PM инкрементально после каждого значимого действия (не только в конце сессии).

### Уровень 3 — По запросу (загружается только когда нужно)
```
knowledge-base/
  PROJECT_STATE.md       ← живой снимок
  INBOX/                 ← зона сброса материалов
    meetings/
    external/
    raw/
  tasks/
    konstantin.md
    kirill.md
    nikita.md
  milestones/
    m1.md
    m2.md
    m3.md
  decisions/
    2026-05.md
  open-questions/
  people/

research/
  INDEX.md               ← оглавление всей базы знаний
  hardware/              ← спецификации браслета, сенсоры
  sdk/                   ← G-Band SDK, методы, API
  metrics/               ← BPM, HRV, SpO2 — как считать
  sources/               ← сырые документы
```

### Защита от потери данных
- **Инкрементальные сохранения** — `PROJECT_STATE.md` обновляется после каждого значимого действия
- **Stop hook** — при штатном завершении Claude Code запускает напоминание сохранить состояние
- **Session lock** — файл `knowledge-base/.session-lock` создаётся при старте, удаляется при чистом завершении. При следующем старте PM проверяет его наличие и предупреждает о незакрытой сессии
- **Obsidian Git** — авто-коммит каждые 15 минут (резервная копия даже при падении VSCode)

## Секция 3. Инструкции агентов (CLAUDE.md)

### Шаблон для всех агентов
```markdown
# [Имя] — [Роль]
## Identity: кто я, что делаю, чего НЕ делаю
## Context: путь к knowledge-base/PROJECT_STATE.md
## Session Start:
  1. Прочитать PROJECT_STATE.md
  2. Glob + Grep по своей папке — что изменилось
  3. Проверить INBOX/ на новые материалы
  4. Прочитать lessons-learned.md
## Boundaries: что трогаю / что не трогаю
## Tools: какие MCP-инструменты использую
```

### Самообучение через lessons-learned.md
У каждого агента в своей папке — файл `lessons-learned.md`:
```markdown
## 2026-05-16 — [Описание ошибки]
Проблема: что пошло не так
Правило: что делать по-другому
```
- `CLAUDE.md` каждого агента обязывает читать `lessons-learned.md` перед началом работы
- PM может добавлять записи в любой `lessons-learned.md`
- Агент сам добавляет записи когда обнаруживает повторяющуюся ошибку

### PM как главный инструктор
В `docs_web/CLAUDE.md` явно прописано:
- PM имеет право читать и обновлять `CLAUDE.md` любого субагента
- PM может усиливать знания агентов, добавляя правила и контекст
- После обновления инструкций — коммит с описанием что и зачем изменено

## Секция 4. Пайплайн загрузки материалов

### Первичная загрузка (разово при инициализации)
| Материал | Куда |
|---|---|
| `meetings/Звонок 12.05.txt` | INBOX/meetings/ → обработка |
| `neiry_pulse_3.html` | knowledge-base/milestones/ |
| `Neiry_Vitro_Metrics_Registry_v0.2.xlsx` | INBOX/raw/ → research/metrics/ |
| PRDs, ТЗ из docs_web | knowledge-base/milestones/ + decisions/ |

### Регулярный workflow
1. **Встреча** → транскрипцию в `INBOX/meetings/` → PM запускает `meeting-transcript-analyst` скилл → tasks, decisions, open-questions обновлены автоматически
2. **Google Doc** → говоришь PM «забери по ссылке» → MCP gdrive скачивает → INBOX/ → обработка
3. **Вопрос по железу** → R&D Researcher исследует через `parallel-cli` → сохраняет в `research/` → объясняет простым языком
4. **Любой артефакт** → `INBOX/raw/` → PM разбирается куда положить

## Секция 5. Делегирование и оркестрация

**Нормальный режим:** ты сам переключаешь терминал между папками агентов.

**Делегирование через PM:** когда PM видит задачу для субагента — запускает его через Agent tool:
```
PM получает задачу "создай wireframe для экрана настроек"
→ PM пишет бриф в UI_assets/BRIEF.md
→ PM запускает субагента в UI_assets/ через Agent tool
→ Субагент выполняет → результат в UI_assets/output/
→ PM забирает результат, обновляет PROJECT_STATE.md
```

**Бриф-файлы** — стандартный способ передачи задач:
- `UI_assets/BRIEF.md` — задание для UX/UI дизайнера
- `dashboard/BRIEF.md` — задание для Frontend Dev
- `research/BRIEF.md` — вопрос для R&D Researcher

## Секция 6. Настройка Obsidian с нуля

### Шаг 1. Открыть vault
`Obsidian → Open another vault → Open folder as vault → NEIRY/knowledge-base/`

Second Brain остаётся отдельным vault — переключение между ними через меню Obsidian (иконка хранилища в левом нижнем углу).

### Шаг 2. Установить плагины (Settings → Community plugins)
| Плагин | Зачем |
|---|---|
| **Dataview** | SQL-запросы по заметкам: «покажи все открытые задачи» |
| **Tasks** | Единый трекер задач с дедлайнами прямо в заметках |
| **Templater** | Шаблоны для встреч, решений, задач |
| **Calendar** | Timeline вид — активность по дням |
| **Obsidian Git** | Авто-коммит каждые 15 минут — защита от потери данных |

### Шаг 3. Инициализировать git в knowledge-base/
```bash
cd NEIRY/knowledge-base && git init && git add . && git commit -m "Init knowledge base"
```
Obsidian Git требует отдельный git-репозиторий внутри vault. `docs_web/` — отдельный репо, `knowledge-base/` — отдельный репо.

### Шаг 4. Настроить Obsidian Git
`Settings → Obsidian Git → Auto backup interval: 15` (минут)

### Шаг 5. Структура создаётся агентом
После инициализации системы Obsidian увидит все папки автоматически.

### Шаг 6. .gitignore
Добавить в `.gitignore`:
```
knowledge-base/.obsidian/workspace*
knowledge-base/.obsidian/plugins/
.superpowers/
```
Сами заметки (`.md` файлы) — версионируются.

## Скоуп реализации

### Создать с нуля
- `NEIRY/knowledge-base/` — Obsidian vault + все папки + `PROJECT_STATE.md` + шаблоны
- `NEIRY/research/` — папка R&D агента + `CLAUDE.md` + `INDEX.md` + структура подпапок
- `UI_assets/CLAUDE.md` — инструкции UX/UI дизайнера
- `lessons-learned.md` для каждого агента
- Stop hook в Claude Code settings

### Обновить существующее
- `docs_web/CLAUDE.md` — добавить: путь к knowledge-base, ритуал старта сессии, права обновлять других агентов, session-lock механизм
- `dashboard/CLAUDE.md` — добавить: BRIEF.md workflow, ссылку на knowledge-base

### Первичная загрузка материалов
- Обработать `meetings/Звонок 12.05.txt` через meeting-transcript-analyst
- Обработать `Neiry_Vitro_Metrics_Registry_v0.2.xlsx`
- Перенести данные из `neiry_pulse_3.html` в milestones/

### Не входит в скоуп
- Изменения в `mobile/` и `backend/`
- Настройка Obsidian плагинов (ручная, 5 минут по инструкции выше)
- Создание аккаунтов или настройка внешних сервисов

## Технические ограничения

- Максимальное контекстное окно Claude — **200k токенов** (не 400k). Трёхуровневая память решает это архитектурно: в контексте одновременно ~20-50k токенов.
- MCP-интеграции (gdrive, google-sheets, gmail, calendar) уже настроены — используются по запросу, не непрерывно.
- `parallel-cli` (~/.local/bin/parallel-cli) авторизован — используется R&D Researcher для веб-поиска.
