## Открытые риски — подробно

### 🟡 R-009. App Store review занимает больше 7 дней

**Категория:** Юр / Compliance · **Milestone:** Ф1 release · **Owner:** PM · **Статус:** in-mitigation

**Что:** Apple ревью health-приложений длится непредсказуемо: типичный срок 1–7 дней, но для приложений с health-данными бывает до 14 дней. После сдвига Ф1 release на 23.06 буфер до пилотов 24.06 — всего 1 день. Если Apple отклонит билд или пришлёт замечания, iOS-версия не успеет стартовать одновременно с пилотами, и придётся запускать пилоты только на Android.

**Что нужно сделать:** Залить «прото-бандл» в App Store как можно раньше (в течение 23.06), не дожидаясь полной готовности всех экранов, — это запускает Apple-ревью параллельно с финальной шлифовкой. Финальный билд обновлять через update до 24.06. PM мониторит статус ревью в App Store Connect ежедневно, при rejection — оперативно дорабатывает спорные пункты и пересылает.

**Резерв:** Если Apple отказывает или тянет — пилоты 24.06 стартуют Android-only, iOS подключается по мере одобрения ревью.

### 🟡 R-020. Backend baseline pipeline + Калибровка re-trigger

**Категория:** Технические · **Milestone:** Ф1 + pilots · **Owner:** Кирилл · **Статус:** open

**Что:** Backend pipeline для расчёта «личной нормы» HRV — ключевой движок MVP. Включает два сценария: (1) первичная калибровка после онбординга — сбор R-R интервалов 36–48 часов, расчёт hrv_personal_norm, триггер push «готово»; (2) re-trigger из настроек — пользователь нажимает «Сбросить и начать заново», система очищает старый baseline, запускает новый сбор. Без pipeline пользователь получает приложение, в котором ключевая фича «личная норма» не работает — это полностью обесценивает MVP.

**Что нужно сделать:** Кирилл к 23.06 release выкатывает phase 1 (первичная калибровка + ready trigger через FCM); re-trigger flow допиливает к старту пилотов 24.06 (на экране «Калибровка браслета» в настройках UI пользователю обещает работающее поведение). Спецификация формулы HRV — зависимость от Дины (R-021), backend пишется на mock-формуле RMSSD до финального решения, заменяется одной строкой кода.

**Резерв:** Если phase 1 не готов к 23.06 — baseline берётся из научных «средних значений по возрасту/полу» (как Sleep tracking в демо-режиме), реальный персональный расчёт включается через update в первую неделю пилотов. Если re-trigger не готов к 24.06 — CTA «Сбросить» на экране настроек в disabled-состоянии с tooltip «доступно с обновления».

### 🟡 R-021. Дина: формула HRV не выбрана (RMSSD / SDNN / pNN50)

**Что:** Формула расчёта HRV (выбор между RMSSD, SDNN или pNN50) — это спецификация для core-движка приложения. Дина получила сырьё R-R от Кирилла 18.06, но финальная формула до сих пор не зафиксирована. Без выбора Кирилл пишет baseline pipeline на mock-формуле RMSSD (industry standard), но реальная замена возможна только после решения Дины — иначе все исторические HRV-значения пользователей придётся пересчитывать ретроспективно.

**Что нужно сделать:** Дина к 23.06 (вт sync с Костей в 15:00) представляет промежуточный анализ сырых данных + рекомендацию формулы с обоснованием. Если до 24.06 (старт пилотов) формула не финализирована — стартуем пилоты на RMSSD по умолчанию (industry standard для wearables) с пометкой «v0», ретроактивно пересчитываем исторические значения после окончательного выбора. Финальное решение — не позднее 30.06 (Ф2 release).

**Резерв:** RMSSD как safe default — Whoop, Garmin, Apple Watch используют именно его. Риск замены минимален.

**Связано:** R-016 (композитная метрика валидируется Диной), R-020 (Backend baseline pipeline ждёт формулу).

### 🟡 R-022. VIGOR парк недостаточен для Райф-пилота

**Категория:** Логистика · **Milestone:** pilots 24.06 · **Owner:** Алексей · **Статус:** open

**Что:** Райф-пилот требует 13–15 браслетов VIGOR — это premium-модель с 9-осевым IMU и GPS-чипом, нужным для tracking операторов колл-центра в рабочее время. На 08.06 в наличии было только 4 VIGOR (3 на складе + 1 у Кирилла). Если к 24.06 не поступит дополнительная партия — Райф стартует с дефицитом 9–11 устройств.

**Что нужно сделать:** Алексей до 23.06 проверяет фактическую availability VIGOR на складе (могли поступить с момента последней инвентаризации 08.06). Если дефицит подтверждается — эскалация Никите, который связывается напрямую с Veepoo для ускорения поставки. Параллельно Алексей готовит fallback на VITRO (базовая модель, без GPS, но HRV работает) для части операторов — методология меняется, но пилот не блокируется.

**Резерв:** Райф-пилот стартует на смешанном парке (4 VIGOR + 9–11 VITRO). GPS-tracking операторов реализуется только для тех, у кого VIGOR, остальные участвуют без этой метрики. Mine Tracker integration (R-015) пересматривается под смешанный парк.

**Связано:** R-015 (Sync браслет ↔ Mine Tracker), R-018 (буфер сроков).

### 🟡 R-023. BLE-экспертиза Кирилла под сомнением — резерв через Mind Tracker

**Категория:** Технические · **Milestone:** Ф1 + Ф2 · **Owner:** Кирилл + Саша Дымов · **Статус:** open (NEW 19.06)

**Что:** На звонке 19.06 Никита открыто высказал сомнение в глубине BLE-экспертизы Кирилла, особенно по части потоковой передачи данных в режиме тренировки (HR-realtime + GPS-stream одновременно с background-операциями iOS). BLE через Flutter — известно сложная тема: вызовы спускаются на нативный слой через несколько абстракций, что снижает производительность и стабильность по сравнению с прямой работой с iOS API. Если возникнут проблемы со стабильностью BLE при тестировании тренировки (потеря пакетов, fall-через-background, переподключения) — это блокирует Ф2 (тренировка) и подрывает пользовательский опыт.

**Что нужно сделать:** Костя 23.06 договаривается с Сашей Дымовым (PM Mind Tracker) о консультации команды Mind Tracker по BLE-вопросам — у них накоплен глубокий опыт работы с непрерывной BLE-потоковой передачей ЭЭГ.
