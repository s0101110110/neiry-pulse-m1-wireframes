Viewport-fit паттерн (общий)

Обе kiosk-страницы рендерятся в фиксированном canvas 1920×1080, который скейлится под viewport через CSS transform: scale(...). Это гарантирует идентичный рендер на ноуте разработчика, MacBook 13/14/15, плазме 42″ — независимо от browser zoom и DPI.

html, body { margin: 0; padding: 0; width: 100vw; height: 100vh; overflow: hidden; }
body { display: flex; align-items: center; justify-content: center; }
#kiosk-stage {
  width: 1920px; height: 1080px;
  flex-shrink: 0;
  transform-origin: center center;
  position: relative; overflow: hidden;
  background: hsl(var(--background));
}

// JS:
function fitStage() {
  const scale = Math.min(window.innerWidth / 1920, window.innerHeight / 1080);
  document.getElementById('kiosk-stage').style.transform = `scale(${scale})`;
}
fitStage();
window.addEventListener('resize', fitStage);
Важно для React-портирования: весь kiosk-контент должен оставаться внутри обёртки #kiosk-stage. Модалки (fixed inset-0) внутри scaled-ancestor ведут себя как absolute относительно stage — это норма, claim-overlay и QR-модалки уже так работают.
Dev-bar (нижняя плашка с кнопками state/persona) — служебная, для QA/демонстрации. Находится снаружи #kiosk-stage (не масштабируется). На проде её можно скрыть или удалить — никакого продакшен-кода она не несёт.

kiosk-v2.html — стена 3 карточек

Главный экран на плазме. 3 одинаковые карточки участников (Костя, Алексей, Гость) идут в ряд. Каждая показывает BPM + мини-ECG + NSI gauge. Клик по карточке открывает детальный профайл kiosk-drilldown.html в новой вкладке через query-параметр.

4.1 State machine

Глобальный state стены управляется через атрибут data-state на <body>. CSS-селекторы показывают/скрывают слоты внутри карточек.

data-stateЧто показываетсяКогда
idle Все 3 карточки в режиме «надень браслет» (пульсирующее сердце + копия) Стартовое состояние, никто не подключён
active-2 2 карточки активны (Костя, Алексей), 3-я ждёт калибровки гостя Якоря на стенде уже надели браслеты, гость ещё нет
active-3 Все 3 карточки активны Гость подключился, идёт калибровка / сессия
pending-claim QR-модалка поверх стены — забрать отчёт Гость закончил тест Баевского, открыт QR на телефон 
// Переключение state:
document.body.setAttribute('data-state', 'active-3');

// CSS реагирует декларативно:
body[data-state="active-3"] .when-active { display: flex; }
body[data-state="idle"]     .when-idle   { display: flex; }

4.4 JS API

ФункцияЧто делает

tickBPM()Sin-волна вокруг baseBPMs — вызывается каждые 1500 мс
tickNSI()Sin-волна NSI — каждые 200 мс, разные фазы для каждой персоны

Управление с клавиатуры (служебное, для QA): 1 = idle, 2 = active-2, 3 = active-3, 4 = pending-claim, Esc = закрыть QR.

4.5 Что нужно от backend для kiosk-v2

Описано понятным языком — конкретный API-контракт согласовываем отдельно.

BPM (поток): текущий пульс для каждой из 3 персон. Обновление 1–2 раза в секунду. Лучше всего через WebSocket или Server-Sent Events. Если REST — поллинг каждые 1.5 сек по 3 endpoint'ам одновременно (или один батч-эндпоинт).
NSI / Stress (поток): индекс стресса 0–100 для каждой персоны. Может обновляться реже — раз в 5–10 сек хватит (визуально мы шумом sin-волны добавляем «жизнь»). Алгоритм считать на бэке — отдавать готовое число.
ECG (поток): отдельная история. Сейчас рисуется параметрический P-QRS-T-pattern на основе BPM. Если есть реальный сэмплированный сигнал с браслета — можно либо отдать массив точек (≥100 Гц), либо оставить наш генератор и использовать только BPM (визуально неотличимо для зрителя).
BLE-статус (события): событие «браслет подключился / потерял связь / отключился» для каждой персоны. Просто строка connected / lost / disconnected — кладём в data-ble.
Калибровка гостя (событие): старт/прогресс/конец калибровки. Сейчас 12-секундный счётчик имитирует процесс. На бэке нужно сообщить «гость готов к измерениям» — после этого переключаем data-state на active-3.
Готовый отчёт (событие): когда гость закончил тест Баевского и можно показать QR. Бэк генерирует короткую ссылку → передаёт в #qr-modal. Состояние переключаем на pending-claim.

kiosk-drilldown.html — детальный профайл

Открывается по клику на карточку стены kiosk-v2 через URL kiosk-drilldown.html?id=kostya. Показывает живые данные браслета + предиктивные метрики + Баевский тест с пошаговым прохождением.

5.1 URL-параметры

Все параметры — для демонстрации и QA, не часть продакшен-API. На проде ?id= остаётся (определяет персону), остальное — служебное.

ПараметрЗначенияЧто делает
?id= kostya alexey guest Какую персону показать. По умолчанию guest.
?state= empty drilldown phase-1 phase-2 phase-3 result claim Принудительно стартовать в указанном состоянии. Удобно для скриншотов / тестов.
?session=active active Если задано — открывается в drilldown, иначе в empty.
?imu= rest light motion off Принудительно выставить статус IMU (для проверки 4 состояний и блокировки кнопки теста).

5.6 Что нужно от backend для drilldown

1. Снимок текущего состояния персоны: при открытии страницы нужен полный объект из 5.4 — BPM, Stress, IMU, метрики, Sleep, Баевский. Один REST-вызов на старте, шаблон ответа уже описан в MOCK_DATA.
2. Live-обновления BPM + ECG: WebSocket / SSE, 1–2 обновления в секунду на BPM, ECG — либо реальный сигнал ≥100 Гц, либо оставить наш генератор (визуально неотличимо).
3. Stress (NSI): обновление 1 раз в 5–10 сек хватит. Бэк отдаёт уже посчитанное число 0–100 + лейбл зоны.
4. IMU-статус: событие смены состояния (rest → light → motion → off). Сейчас сама блокировка кнопки Теста завязана на этот статус — если IMU «не в покое», тест запустить нельзя.
5. Предиктивные метрики (recovery / burnout / drowsy): обновляются редко (раз в минуту / раз за сессию). Один REST-эндпоинт, дёргать по таймеру или после ключевых событий (закончил тест, новые данные сна).
6. Sleep: данные за прошлую ночь — приходят один раз при загрузке страницы. Структура: длительность + % фаз (DEEP / REM / LIGHT).
7. Баевский тест: отдельный flow. Сейчас имитируется через клик и таймеры:
  • Старт теста (фаза 1 — покой 5 мин): бэк начинает запись R-R интервалов
  • Переход на фазу 2 (STROOP на планшете): на стороне UI открывается экран «передайте планшет», результат STROOP'а нужен бэку для расчёта ИН2
  • Фаза 3 (нагрузка): браслет продолжает писать, UI просто показывает таймер
  • Конец теста: бэк отдаёт ИН Σ, 3 фазы (label / ИН / вердикт), общий вердикт
  • После закрытия модалки результата — генерится QR-ссылка для claim
Empty-state: если бэк не вернул данные (нет активной сессии) — UI просто показывает skeleton. Никакого дополнительного запроса не нужно — добавь класс body.state-empty и всё схлопнется в плейсхолдеры.
06

Общие правила (важно для обеих страниц)

  • State-driven CSS: видимость блоков переключается через атрибут/класс на <body> и CSS-селекторы вида body.state-X .when-X { display: flex; }. Не делать видимость через JS — всё декларативно, отлаживается за секунду в DevTools.
  • Цвета — только через токены: hsl(var(--success)), прозрачность через hsl(var(--token) / 0.18). Хардкод hex/rgb запрещён — при смене темы всё сломается.
  • Tabular-nums: для всех live-чисел добавлять font-variant-numeric: tabular-nums, иначе цифры «прыгают» при смене значения.
  • 4-pixel grid: все отступы и размеры — кратны 4. Если хочется поставить 18 — округлять до 16 или 20.
  • SVG-gauge: на каждой <circle> / <path> для арок ставится pathLength="100". Тогда stroke-dashoffset = 100 − value% — без подсчёта окружности.
  • ECG-генератор: параметрический P-QRS-T waveform, привязан к BPM. Длительности компонентов масштабируются при высоком пульсе. Если есть реальный сэмплированный ECG с браслета — можно просто кормить его в полилинию SVG, генератор не нужен.
07

Что НЕ менять

Не убирать viewport-fit pattern. Плазма 42″ не рендерится корректно без transform: scale() — обрезается на 100% zoom, растягивается на 75%. Если делаешь рефакторинг — оставляй #kiosk-stage + fitStage().
Не менять структуру SVG-gauge (NSI, Stress, Sleep, Recovery-bar). Все они завязаны на pathLength="100" и stroke-dashoffset. Изменение viewBox / радиуса сломает математику процентов.
Dev-bar pills (нижняя плашка) — служебные. Можно скрывать на проде через CSS (.dev-bar { display: none }), но не удалять обработчики событий — на них завязаны хоткеи и QA-сценарии. Лучше оставить под флагом ?dev=1.
Не переименовывать существующие DOM-id. Все они задокументированы в этом файле и используются как контракт между UI и логикой обновления. Если нужен новый id — добавлять, не переименовывать.
Не вводить новые font-size инлайн. Использовать --t-caption / --t-body / ... — список из 6 ступеней (7 с --t-widget). Если нужен новый размер — добавлять в :root как ступень, не хардкодить пиксели.
Шрифты — 3 семейства, не больше. Onest / Space Grotesk / Geist Mono. Не добавлять новые без согласования с дизайнером.
