## Задача 1 (вся суббота) — m3-w1-1: Mode enum + экран выбора режима

### Что делаем

Двухуровневый выбор роли при онбординге, который маршрутизирует пользователя в Sport-флоу или Corporate-флоу. Этот enum пронизывает весь app lifetime: дашборды, алерты, экспорты, фильтры.

### Скоуп

**Backend:**

- Enum `UserMode` с двумя уровнями:
  - L1: `SPORT` | `CORPORATE`
  - L2 (только для Corporate): `OFFICE` | `DRIVER` | `CONDUCTOR` | `DISPATCHER`
- Миграция БД: поле `mode` + `sub_role` в таблице users
- Auth API возвращает `mode` в ответе на `/login` и `/me`
- Endpoint `POST /users/me/mode` для установки/смены режима

**Mobile:**

- Экран после регистрации: «Какая у вас роль?» с двумя крупными карточками (Sport / Corporate)
- При выборе Corporate — второй экран с четырьмя ролями (office / водитель / машинист / диспетчер)
- Краткое описание под каждой ролью (1 строка: какие метрики увидит)
- Сохранение выбора локально + синк на бэк

### Acceptance criteria

- [ ] Auth API отдаёт `mode` и `sub_role` в JWT/ответе
- [ ] Схема БД мигрирует без потери данных существующих юзеров (дефолт = null, заставляем выбрать при первом входе)
- [ ] Мобильный экран отображается ровно один раз после регистрации
- [ ] Можно сменить режим в настройках (для дебага и пилотов)
- [ ] Enum-значения готовы к использованию из w2-1 (Sport ветка) и w3-1 (Corporate ветка)

### Что Mode enum должен включить дальше

- **Sport** → m3-w2-1: тренировки, GPS, пульсовые зоны, лидерборд
- **Corporate / Office** → дашборд стресс / восстановление / выгорание
- **Corporate / Driver | Conductor | Dispatcher** → дашборд + алерты сонливости (m3-w3-2)

### Что НЕ делаем сейчас

- Не реализуем сами фичи Sport/Corporate — только маршрутизацию
- Не делаем UI consent/privacy (это w2-6, ждёт дизайн от w2-7a)

### Риски

- **R-001** (sleep data) — косвенно: если Corporate выберут водители, а сон не считается, доверие падает
- **R-004** (sport pilot) — если sport-роль никто не выбирает на пилоте, демо лиги становится критичным

### Что показать Никите в понедельник

PR + скриншоты двух экранов выбора + демо в эмуляторе: новый юзер → выбирает роль → попадает на заглушку соответствующего дашборда.

## Задача 2 (воскресенье, первая половина дня) — m3-w1-7: FCM-инфра

### Что делаем

Поднять Firebase Cloud Messaging как доставку push для двух будущих сценариев: safety-алерты сонливости (W3) и sport-напоминания (W4). Сейчас — только трубы, без бизнес-логики.

### Скоуп

**Firebase:**

- Создать проект `neiry-pulse-prod` (и `-dev` если нужно)
- Сгенерировать service account JSON для бэкенда
- Подключить Android app (package name, SHA-1)
- Сохранить ключи в защищённом месте (не в git!)

**Backend:**

- Сервис `PushDispatcher` с методом `send(user_id, payload)`
- Таблица `device_tokens` (user_id, fcm_token, platform, updated_at)
- Endpoint `POST /devices/register-token` для приёма токена с устройства

**Mobile (Android):**

- Подключить FCM SDK
- Получить токен на старте app, отправить на бэк
- Обработчик входящих push (минимальный: показать notification)

### Acceptance criteria

- [ ] Firebase проект создан, ключи у Кирилла + бэкап
- [ ] Тестовое Android-устройство получает push через `PushDispatcher.send()` с бэкенда
- [ ] Токен устройства сохраняется в БД при регистрации/логине
- [ ] Готово к интеграции в w4-1 (safety-алерты + sport-reminders)

### Что НЕ делаем

- Не пишем логику алертов сонливости (это w3-2)
- Не пишем тексты уведомлений (брифинг от Никиты позже)
- Не делаем iOS-часть (это в w3-6 после iOS Bridge)

### Риски

- **R-003** (push не успевает к W4) — раннее поднятие в W1 даёт буфер 3 недели
- **R-006** (152-ФЗ) — push-уведомления = persdata, нужно consent. На этом этапе только инфра, consent UI делается в w2-6

### Что показать Никите

Скрин Firebase console + видео 5 сек: с бэкенда `curl POST → notification всплывает на тестовом устройстве`. Решение для обсуждения: успеваем ли с safety-алертами к W4, нужен ли промежуточный mock-релиз.

## Задача 3 (воскресенье, вторая половина) — m3-w2-3: Мок-сезон демо-данных лиги

### Что делаем

JSON-фикстуры с 10–15 синтетическими спортсменами на 2 недели мок-сезона, чтобы дашборд `league.neiry.ru` выглядел живым на демо/инвесторских питчах без ожидания реального пилота.

### Скоуп

**Данные (генерация):**

- 10–15 профилей спортсменов: имя, аватар (placeholder), возраст, дисциплина (бег / триатлон / велик)
- На каждого: минимум 2 недели тренировок (~3–5 сессий/неделю)
- Каждая сессия: дата, тип активности, длительность, GPS-трек (можно полилинию-заглушку), HR-зоны (по возрасту), дистанция, темп
- Композитный Fitness Score (пока R&D-1 не дала формулу — используем placeholder: weighted sum HRV + load + recovery)

**Структура:**

- JSON-файлы по схеме, совпадающей с output Sport Mode (m3-w2-1)
- Сидинг-скрипт для загрузки в БД лиги
- Реалистичный разброс: 2–3 топовых, основная масса в середине, 2 отстающих — для красивого лидерборда

### Acceptance criteria

- [ ] 10–15 профилей в БД + 2+ недели истории на каждого
- [ ] Лидерборд на `league.neiry.ru` рендерится с рейтингом и трендами
- [ ] Графики прогресса не пустые, динамика выглядит правдоподобно
- [ ] Схема данных совместима со Sport Mode (когда тот зальёт реальные данные — без миграций)

### Зависимости и оговорки

- **Fitness Score формула в R&D-1** ещё не финализирована → используем placeholder, маркируем `// TODO: заменить на реальную формулу из R&D-1`
- Не путать с реальным пилотом — это **демо-данные** для питчей

### Риски

- **R-004** (sport pilot молчит) — митигируем: без живых атлетов есть на чём показывать продукт
- **R-001** (sleep сломан) — если Fitness Score включает сон, демо-цифры могут разойтись с реальностью когда починим

### Что показать Никите

Открыть `league.neiry.ru` в браузере, прокликать лидерборд → профиль спортсмена → графики прогресса. Решения для обсуждения: какие поля показывать в карточке спортсмена, какие сегменты разбивки (возраст / дисциплина), что писать в шапке лиги.
