## 1. Что это вообще такое

**DESIGN.md** — открытый стандарт от Google Labs (вырос из проекта Stitch) для описания дизайн-системы продукта **одним текстовым файлом**. Структурированный markdown с YAML-фронтматтером, который читается одновременно человеком, дизайн-инструментом и AI-агентом.

Идея простая:

> Вместо Figma-библиотеки, Notion-страницы с токенами, Storybook'а и отдельного `tokens.json` — **один файл в репо**, который является источником истины для цветов, шрифтов, отступов и компонентов.

Аналогия: как `package.json` описывает зависимости, `DESIGN.md` описывает дизайн-систему.

---

## 2. Зачем это нам

Три причины, по которым мы внедряем DESIGN.md в проекте:

1. **Один источник истины.** Сейчас токены живут в `wireframes/m2/ui-kit.html` (CSS-переменные), в head'ах дизайнера и в Tailwind-конфиге. DESIGN.md консолидирует это.

2. **Машиночитаемость.** AI-агенты (Claude Code, Cursor, Copilot) парсят DESIGN.md и не выдумывают цвета. Когда ты говоришь агенту «сделай кнопку primary» — он смотрит в DESIGN.md, видит `{colors.primary}` и не использует `#FF00AA` из головы.

3. **Защита бренда.** В файле жёстко прописано: wine `#831843` — единственный chromatic accent, не менять. Любой инструмент, читающий DESIGN.md, обязан соблюдать это правило.

## 3. Структура файла

DESIGN.md состоит из двух частей:

### A. YAML-фронтматтер (между `---`)

Машиночитаемые токены. То, что инструменты импортируют автоматически:

```yaml
---
version: alpha
name: Neiry-Pulse-design-analysis
description: "..."

colors:
  primary: "hsl(336 76% 33%)"      # wine #831843
  bio-pulse: "hsl(336 76% 45%)"
  ...

typography:
  display-kiosk:
    fontFamily: "Onest, Space Grotesk, system-ui, sans-serif"
    fontSize: 180px
    fontWeight: 600
    letterSpacing: -6px

rounded:
  md: 8px
  lg: 12px

spacing:
  4: 16px
  6: 24px

components:
  button-primary:
    backgroundColor: "{colors.primary}"
    typography: "{typography.button}"
    rounded: "{rounded.md}"
---
```

**Ключевое:** ссылки вида `{colors.primary}` — это переменные. Один цвет меняется в одном месте, и все компоненты, ссылающиеся на него, обновляются автоматически.

### B. Markdown-секции (после `---`)

Человекочитаемые разделы для контекста:

- **Overview** — философия системы, на чём построена.
- **Colors** — описание ролей каждого цвета (когда использовать wine, когда muted).
- **Typography** — таблица иерархии шрифтов и принципы (например, «tabular nums везде, где цифра живая»).
- **Layout** — сетка, отступы, grid.
- **Components** — описание каждого компонента словами.
- **Principles** — правила («wine не менять», «hairline borders вместо shadows»).
- **Anti-patterns** — чего никогда не делать.

## 4. Как этим пользоваться в верстке

### Сценарий 1 — пишешь компонент с нуля

1. Открываешь `docs_web/DESIGN.md`.
2. Ищешь компонент в секции `components:` фронтматтера или в markdown-описании.
3. Берёшь токены: `backgroundColor: {colors.primary}`, `padding: 10px 16px`, `rounded: {rounded.md}`.
4. Транслируешь в CSS / Tailwind:

```css
.btn-primary {
  background: hsl(336 76% 33%);    /* {colors.primary} */
  color: hsl(0 0% 98%);             /* {colors.primary-foreground} */
  padding: 10px 16px;
  border-radius: 8px;               /* {rounded.md} */
  font: 500 14px/1.2 'Space Grotesk', system-ui, sans-serif;
}
```

или Tailwind:

```html
<button class="bg-[hsl(336_76%_33%)] text-white px-4 py-2.5 rounded-md font-medium text-sm">
  Начать тест
</button>
```

### Сценарий 2 — спрашиваешь AI-агента

Достаточно сказать: **«Сделай KPI-карточку по DESIGN.md»** — агент сам найдёт `dashboard-kpi-card` во фронтматтере и соберёт верстку с правильными токенами.

### Сценарий 3 — нужна тёмная тема

Все цвета определены и для dark, и для light (`primary-dark`, `card-dark`, `border-dark`). Переключение темы — это смена набора CSS-переменных, токены те же.

## 5. Что НЕ делать

❌ **Не использовать хардкод-цвета в коде.** Если в верстке появилось `#A91B5C` (близкий к wine, но не он) — это ошибка. Берёшь только `{colors.primary}`.

❌ **Не править DESIGN.md без согласования с PM.** Файл — источник истины, любая правка ломает все компоненты, которые на него ссылаются.

❌ **Не игнорировать `tabular-nums` на метриках.** Любая цифра, которая обновляется в реальном времени (пульс, HRV, NSI, секунды) — должна быть Geist Mono с `font-variant-numeric: tabular-nums`. Иначе цифры прыгают при смене 78→79.

❌ **Не вводить вторые акцентные цвета** ради «премиума» или «дизайнерского жеста». Wine — единственный chromatic accent. Bio-палитра (зелёный/жёлтый/красный для NSI) — служебная, привязана к семантике стресс-зон.

## 6. Файлы в репо, связанные с DESIGN.md

| Файл | Что внутри |
|---|---|
| `docs_web/DESIGN.md` | Источник истины — токены + принципы |
| `docs_web/DESIGN-preview.html` | Визуальное превью DESIGN.md в браузере (откроется как обычная страница) |
| `wireframes/m2/ui-kit.html` | Канонические CSS-переменные, синхронизированы с DESIGN.md |
| `wireframes/m2/*.html` | Готовые экраны kiosk/dashboard/mobile, используют те же токены |

**Когда что открывать:**
- Хочешь **увидеть** дизайн-систему → `DESIGN-preview.html` (живые компоненты в браузере).
- Хочешь **спецификацию** → `DESIGN.md` (текст).
- Хочешь **CSS-переменные для копирования** → `wireframes/m2/ui-kit.html`.

## 7. Стек, который ждёт DESIGN.md

Когда будем переезжать с HTML-вайрфреймов на React:

- **Стек:** Next.js + Tailwind CSS + shadcn/ui (это то, под что заточены наши токены — HSL-формат shadcn).
- **Дашборд:** добавим Tremor для биометрических виджетов поверх shadcn.
- **Перевод DESIGN.md → Tailwind config:** официальный CLI `npx @google/design.md export tailwind` генерирует `tailwind.config.ts` из фронтматтера. Это упростит миграцию.

## 8. Quick reference — самое важное в одной картинке

```
DESIGN.md
├── YAML frontmatter   ← машиночитаемые токены (colors, typography, spacing, components)
│   └── читают: AI-агенты, CLI, Tailwind config generator
└── Markdown sections  ← человекочитаемый контекст (overview, principles, anti-patterns)
    └── читают: разработчик, дизайнер, PM
```

**Один файл = один источник истины.**
