06Синхронизация со смежными командами

 Mindtracker (Headband) и Stim-X (tVNS) — продукты той же организации. У них уже есть наработанная инфраструктура и компоненты, которые можно переиспользовать. Это критично для срока.

Задача · Неделя 1 · Owner: Константин

Цель: понять, что переиспользуем, что строим с нуля

 Провести встречи с командами Mindtracker и Stim-X. Цель — выяснить, какие компоненты у них уже работают, и согласовать, что можно использовать в Neiry Pulse. Результат — документ с inventory и согласованным планом интеграции.

Что выясняем на встречах:

 Регистрация и auth
 Есть ли единый Neiry Account? Можно ли подключиться через SSO/OAuth?

 Профиль пользователя
 Какие поля общие? Где хранится (БД, схема)? Можно ли переиспользовать?

 API контракты телеметрии
 Есть ли единый стандарт передачи биометрики? Можем ли согласовать общую схему?

 Push-сервис
 Один FCM/APNS аккаунт на всю экосистему? Кто управляет ключами?

 Дизайн-система
 Есть ли общий design tokens, компоненты, гайдлайны? Или строим свой?

 Биллинг и подписки
 Общая модель монетизации? Единая платёжная система?

 Стратегия: то, что у Mindtracker / Stim-X уже работает в проде — берём как есть или с минимальной адаптацией. То, чего нет — строим в Neiry Pulse, но в архитектурно совместимом виде, чтобы потом не мигрировать данные.

 Артефакт: документ Neiry_Ecosystem_Sync.md с тремя секциями: «Что переиспользуем», «Что строим с нуля», «Точки совместимости на будущее».

Как работает «Корпорат» одним пресетом

 Один дэшборд, два сценария использования. По умолчанию (роль «Офис») — фокус на выгорании, стрессе, командных агрегатах. Если у юзеров выбрана safety-роль (водитель, машинист, диспетчер) — на overview добавляются виджеты сонливости, утомления и готовности к смене. HR / специалист по охране труда видит то и другое в одном интерфейсе, фильтрует по роли.

Архитектура «гибрид поддоменов»

 league.neiry.ru — отдельный поддомен с брендом лиги. Это нужно для продаж: лига получает «свой» продукт, можно дать white-label под конкретного клиента в Волне 2.

 dashboard.neiry.ru — единая точка входа для корп-сегмента. Технически — та же React-сборка, тот же codebase. Разница — env-конфиг и роутинг NGINX.

12Онбординг

 Самый простой возможный флоу. Mode определяется выбором роли при регистрации.

Шаги первого запуска приложения

Splash + onboarding скрины (2–3 экрана о Neiry Pulse)

Регистрация по email или телефону + пароль

Экран выбора роли (всего 2 варианта в MVP):

— «Я сотрудник компании» → Corporate Mode

— «Я спортсмен» → Sport Mode

Базовый профиль: имя, год рождения, пол, рост, вес

Pairing браслета по BLE (модель определяется автоматически)

Mode-специфичные шаги:

— Sport: максимальный пульс, основной вид спорта (бег / триатлон / вело / лыжи / другое), целевое соревнование

— Corporate: код компании (опционально); ФИО для HR; роль (офис / водитель / машинист / диспетчер) выбирает HR при добавлении

16Риски и митигации

Риск

Не уложимся в 8 недель MVP

Митигация: первым на отказ — push-уведомления и safety-метрики. Sport + офисный корп — must-have.

Риск

iOS BLE-bridge сложнее Android

Митигация: параллельная разработка в Tech Demo. Закроем риск на неделе 2.

Риск

Детектор сонливости требует валидации

Митигация: в MVP — простая модель (ВСР + микро-паттерны). Полная валидация — Волна 2.

Риск

Sport включает много видов спорта

Митигация: в MVP — типизация workout, без специфичных метрик для каждого вида.

Риск

APNS — задержка одобрения от Apple

Митигация: настроить APNS параллельно с iOS-bridge на Tech Demo неделе.

Риск

Sync со смежными командами затягивается

Митигация: жёсткий дедлайн неделя 1. Если не успели — строим своё, согласуем интеграцию в Волне 2.
