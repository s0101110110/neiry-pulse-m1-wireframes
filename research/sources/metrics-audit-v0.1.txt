0. Executive Summary
Этот документ — аудит метрик линейки умных браслетов Neiry Vitro (модели VITRO Base / VANTA Health / VIGOR Sport), построенных на платформе HBandSDK от китайского поставщика Veepoo. Цель — дать Константину как Product Owner рабочий инструмент для решений о монетизации и UI, а Кириллу/разработчикам — спецификацию для имплементации.
Главные выводы
Сенсорная база ограниченная, но достаточная для wellness. Браслет даёт PPG (~25 Hz), IMU 3-ось, термистор, опционально ЭКГ (только VANTA, 512 Hz) и BIA (только VANTA). Этого хватает для пульса, шагов, сна, базового HRV. Не хватает для медицинских решений: - давление без манжеты, - состав тела без 4-электродного BIA, - температура тела (не кожи) — физически невозможны на этом железе.
SDK даёт готовые метрики, доступ к сырому сигналу — открытый вопрос. HBandSDK документирует возможность стриминга сырого PPG (метод readPPGRawData + IPPGRealTimeTransmissionListener), но мы строим аудит исходя из консервативного сценария: сырья нет, работаем с готовыми результатами Veepoo. Это означает: HRV считается их native-библиотекой (libnative-lib.so, метод HRV_LorenzDetection), формула не раскрыта. Полный список вопросов к поставщику — в отдельном документе.
Иерархия метрик из 5 уровней (L0–L4). L0 — сырые сенсорные данные. L1 — первичные метрики (HR, RR, шаги). L2 — производные по международным стандартам (HRV-RMSSD, фазы сна). L3 — композитные индексы (Stress, Recovery — проприетарные). L4 — предиктивные ML-инсайты (Burnout Risk, Injury Risk). Эта структура международно признана и используется WHOOP, Polar, Garmin.
Маркеры доверия — наш главный фильтр против юридических рисков. Из ~14 заявленных Veepoo метрик треть — маркетинговые (глюкоза неинвазивно, состав крови, артериальное давление без калибровки). Они физически невозможны на PPG+BIA-2 железе. Для сегментов Care (пенсионеры) и Tactical (армия) их показ создаёт прямые юридические риски. Рекомендация — премиум-позиционирование «честной биометрии» против китайских конкурентов с 14 функциями.
Архитектура расчётов на 6 слоях. L0/L1 на Hardware (Veepoo MCU). L1/L2 в Edge SDK (HBandSDK + libnative-lib.so). L2/L3 в Backend (FastAPI + Celery). L4 в Analytics layer (Pandas/ML). Это даёт нам контроль над proprietary метриками (L3+) даже при привязке к Veepoo на нижних слоях.
Сравнение с референсами Polar, WHOOP, Garmin. Они показывают меньше метрик, но точнее. Polar Nightly Recharge, WHOOP Recovery, Garmin Body Battery — все строятся на ночном HRV-окне как фундаменте. Мы можем построить аналогичный Neiry Daily Score, если получим доступ к RR-интервалам из SDK (открытый вопрос #1 к поставщику).
1. Методология. Иерархия метрик L0–L4
В индустрии wearables и медтеха используется устоявшаяся иерархия метрик, основанная на расстоянии от сырого сенсорного сигнала и степени проприетарности алгоритма. Эта классификация позволяет: (1) понимать, где можно положиться на международные стандарты, а где мы строим собственную интеллектуальную собственность; (2) определять, на каком слое архитектуры рассчитывается каждая метрика; (3) корректно коммуницировать точность пользователю и регуляторам.
1.1. L0 — Сырые сенсорные данные (Raw Sensor Data)
Необработанные сигналы прямо с АЦП сенсора. На уровне L0 работают DSP-алгоритмы (Digital Signal Processing — цифровая обработка сигналов): фильтрация шума, motion artifact rejection (подавление артефактов движения), полосовая фильтрация.
Где находятся: в основном внутри firmware Veepoo на MCU браслета; частично передаются в Mobile App через SDK (PPG streaming, ECG ADC samples).
Примеры: PPG waveform на 25 Hz (зелёный канал), ECG ADC samples на 512 Hz, акселерометр {x,y,z} на 25 Hz, термистор raw mV.
1.2. L1 — Первичные метрики (Primary / Direct Metrics)
Прямые измерения, извлекаемые из сырого сигнала одним алгоритмом. У Polar это называется Direct Measurements. Большинство L1-метрик международно стандартизованы и не требуют проприетарных алгоритмов.
Где рассчитываются: MCU браслета или SDK на смартфоне.
Примеры: HR (Heart Rate, частота сердечных сокращений), RR-интервалы (Inter-Beat Intervals), SpO₂ (сатурация кислорода), шаги, температура запястья, ECG samples.
1.3. L2 — Производные / составные метрики (Derived / Composite Metrics)
Метрики, рассчитанные из L1 по научно валидированным формулам (Task Force HRV Standards 1996, ESC Guidelines, AHA recommendations). Это всё ещё международные стандарты, формулы открыты и не являются IP.
Где рассчитываются: SDK native lib (если есть готовый алгоритм) или Backend (если мы хотим контроль над формулой).
Примеры: HRV-RMSSD, HRV-SDNN, HRV-pNN50 (все из RR-интервалов), VO₂max (по Firstbeat/ACSM formula), фазы сна (на основе HR + HRV + motion), TRIMP (Training Impulse).
1.4. L3 — Композитные индексы (Composite Scores / Wellness Indices)
«Квази-метрики», как они часто называются в продуктовой разработке. Это проприетарные алгоритмы, где взвешиваются и комбинируются метрики L2. Часто это и есть USP-кандидаты продукта. Формулы обычно публично не раскрываются.
Где рассчитываются: Backend Neiry (расчётный движок) или, для покупных решений, в SDK.
Примеры: WHOOP Recovery Score, Oura Readiness, Polar Nightly Recharge, Garmin Body Battery, Apple Vitals. Наши кандидаты: Neiry Daily Score, Burnout Risk Index, Closed-Loop Recovery Index.
1.5. L4 — Предиктивные и поведенческие инсайты (Predictive / Behavioral Insights)
ML-модели, обученные на исторических данных пользователя (или популяции), которые дают прогнозы и рекомендации, а не измерения.
Где рассчитываются: Analytics layer Neiry (Pandas + scikit-learn / PyTorch), batch-задачи через Celery.
Примеры: Burnout prediction за 2 недели до события, Injury Risk на основе HRV-трендов, Adaptive Training Load, Sleep Need Forecast.
Терминологическая заметка
В научной литературе и документации производителей встречается ряд синонимов для этой иерархии. Polar использует Direct vs Calculated. Apple — Vital Signs vs Wellness Scores. Firstbeat — Physiological Parameters vs Lifestyle Insights. Все эти системы по сути описывают одну и ту же шкалу проприетарности и расстояния от сенсора. Мы используем уровни L0–L4 как наиболее universal и совместимое с архитектурным мышлением.
2. Архитектурные слои: где какая метрика считается
Иерархия метрик L0–L4 ложится на 6 архитектурных слоёв нашего стека. Понимание, где именно рассчитывается метрика, критично для: производительности (latency), энергопотребления (battery life), приватности (передаём ли мы сырые данные в облако), и контроля (можем ли мы менять формулу без OTA-обновления прошивки).

Стратегические следствия
L0 и L1 — у поставщика. Мы не можем менять то, как Veepoo считает HR или агрегирует шаги в firmware. Это даёт стабильность, но снимает с нас часть контроля.
L2 — пограничная зона. HRV считается их native-библиотекой (HRV_LorenzDetection). Если мы получим доступ к RR-интервалам из SDK, мы можем дублировать расчёт HRV на нашем backend по нашей формуле — и это будет наш ключевой контроль качества (см. раздел 6 о дилеммах).
L3 и L4 — целиком наши. Stress Score, Recovery, Burnout Risk считаются на Neiry Backend. Это наша IP. Stress есть и в SDK от Veepoo (часть OriginData), но мы можем дополнять и переопределять его собственной формулой.

Физические ограничения, которые невозможно обойти
Артериальное давление без манжеты — нельзя точно. Метод PTT (Pulse Transit Time) даёт грубую оценку, FDA одобрила только Samsung Galaxy Watch с обязательной ежемесячной калибровкой манжетой. Наша рекомендация: давление либо за toggle «Дополнительно (экспериментально)» с явным дисклеймером, либо в Private-mode с onboarding-калибровкой эталонной манжетой (как у Samsung).
Глюкоза неинвазивно — невозможна. FDA в 2024 году выпустила официальное предупреждение против неинвазивного измерения глюкозы в носимых устройствах. Apple, Samsung, Google более 10 лет работают над этим — рабочего продукта ни у кого нет. Если поставщик заявляет такую функцию, это маркетинг (🔴) и должно быть удалено из дефолтных пресетов.
Состав крови (мочевая кислота, липиды) — невозможен. Биохимия крови меряется в лаборатории через венозный забор и реактивы. Через свет на запястье эти концентрации детектировать физически невозможно. Это полностью маркетинговая метрика (🔴).
Состав тела — недостоверен. Реальный BIA требует 4 электродов на разных конечностях. Браслет имеет 2 электрода на одной руке — это меряет сопротивление маленького участка предплечья, не состав тела. Метрика 🟠 «Сомнительно».
Температура — это температура кожи, не тела. Термистор измеряет температуру кожи запястья. Разница с core temperature 2–4 °C, зависит от среды и кровотока. В UI рекомендуется называть «температура запястья» (skin temperature), не «температура тела».
4. Качество данных и маркеры доверия
Из ~14 типов мониторинга, заявленных в спецификации поставщика, при продуктовой проверке примерно треть оказались маркетинговыми, не клиническими. Это создаёт юридический и репутационный риск, особенно для сегментов Care (пенсионеры) и Tactical (армия и силовые структуры).

Стратегическая рекомендация
В Mobile App и дашбордах по умолчанию показывать только 🟢 и 🟡 метрики. Метрики 🟠 — с пояснением о точности. Метрики 🔴 — за переключателем «Дополнительно (экспериментально)» с дисклеймером, либо убрать совсем.
Это даёт премиум-позиционирование «честной биометрии» Neiry против китайских конкурентов с 14 функциями подряд (Honor, Huawei, Xiaomi). WHOOP, Oura, Garmin намеренно показывают меньше метрик, но точных. Эта же стратегия работает для нас — особенно в сегментах Corporate (HR-аналитика) и Care (мониторинг пенсионеров), где репутация и юридическая чистота критичнее количества функций.
5. Инвентарь возможностей HBandSDK
Этот раздел — краткое резюме того, что мы знаем о SDK из официальной документации Veepoo (deepwiki.com/HBandSDK/Android_Ble_SDK). Это входит в задачу HBandSDK Inventory Document, которую делает Кирилл, но в контексте аудита метрик мы фиксируем здесь самое важное.
5.1. Центральная точка входа — VPOperateManager
SDK реализует facade-паттерн: одна singleton-точка VPOperateManager, через которую вызываются все операции. Все методы асинхронные, результаты приходят через listener-интерфейсы.
5.2. Real-time vital signs — взаимоисключающие операции
Критическое ограничение SDK: одновременно может выполняться только ОДНА measurement-операция. HR / BP / SpO₂ / Temperature — взаимоисключающие. Старт BP блокирует HR и наоборот. Это требует queue-логики в Mobile App.
API-методы и слушатели:
startDetectHeart() → IHeartDataListener — пульс, 30–200 bpm
startDetectBP() → IBPDetectDataListener — давление, требует 50–60 сек
startDetectSpo2h() → ISpo2hDataListener — SpO₂
startDetectTempture() → ITemptureDetectDataListener — температура запястья
5.3. ECG — sample-level доступ
Для модели VANTA доступен поток ADC сырых данных ЭКГ. Native-обработка (notch filter 50/60 Hz, R-peak detection) выполняется в libnative-lib.so на смартфоне:
JNIChange.ecgFilter(...) → App_EcgNotchFilter — фильтрация powerline
JNIChange.ecgDetectEntry(...) → Ecg_DetectEntry — детекция QRS
Частота: 512 Hz, длительность сессии: 30 секунд
5.4. PPG raw — потенциально доступен
Согласно документации, SDK поддерживает два режима стриминга сырого PPG:
MODE1 — каждые 15 минут по 10 секунд (low-power)
MODE2 — каждые 5 минут по 60 секунд (high-resolution)
Listener: IPPGRealTimeTransmissionListener.onGreenLightDataReport(List<Integer>) — зелёный свет на 25 Hz
Синхронно: onAccelerationDataReport(List<AccelerationData>) — для motion artifact rejection
Открытый вопрос #1 к поставщику: Подтверждается ли стриминг сырого PPG в наших прошивках VITRO/VANTA/VIGOR? Есть ли ограничения по battery life? Это критично для решения, можем ли мы считать собственный HRV.
5.5. HRV — native Lorenz plot
Расчёт HRV выполняется в native-библиотеке через метод HRV_LorenzDetection. Это стандартная техника анализа Лоренца / Пуанкаре: точки (RR(n), RR(n+1)) на плоскости с эллипсом SD1/SD2. Метод научно валидирован, но Veepoo не раскрывает состав возвращаемой структуры bod_hrv_char_t.
Открытый вопрос #2 к поставщику: Какие именно поля содержит структура bod_hrv_char_t? Возвращается ли RMSSD, SDNN, pNN50, SD1/SD2, LF/HF? Без этой документации мы либо завязаны на одну метрику (вероятно SD1), либо вынуждены считать всё сами из RR.
5.6. Mini Checkup — комплексная оценка
SDK предоставляет отдельный API «Mini Checkup» — одноразовый запуск 10 параметров health assessment. Деталей в публичной документации мало; нужно изучить через Кирилла на этапе SDK Inventory. Это интересный USP-кандидат для премиум-пресета (VANTA + Recovery центры).
5.7. Historical sync — sleep, steps, activity
Все агрегированные метрики (фазы сна, шаги, активные минуты) хранятся на устройстве и синхронизируются batch-методом раз в N минут через дополнительные API (раздел 4.7 SDK).
6. Стратегические дилеммы: SDK vs собственный расчёт
На основе аудита возникают четыре ключевые продуктовые дилеммы. Каждая требует решения Константина — они влияют на архитектуру, скорость разработки и конкурентное позиционирование.
Дилемма 1. HRV — брать готовый Veepoo или считать свой?
Сценарий A — Готовый HRV от Veepoo
Плюсы: ничего не делаем, быстрый Time-to-Market. Считается на устройстве — энерго-эффективно.
Минусы: формула не раскрыта (bod_hrv_char_t). Привязаны к Veepoo. Если поставщик поменяет алгоритм в OTA-обновлении, наши данные исторически несравнимы.
Маркер: 🟡 Условно (зависит от качества peak detection Veepoo).
Сценарий B — Собственный HRV на наших RR
Плюсы: контроль формулы. Можем считать любые HRV-метрики (RMSSD, SDNN, pNN50, SD1/SD2, LF/HF, всё что нужно). Возможность дать научно-валидированный результат сравнимый с Polar.
Минусы: требует доступа к RR из SDK (открытый вопрос). Нагрузка на backend. Energy cost увеличится из-за стриминга RR.
Маркер: 🟢 Достоверно (если PPG-точность peak detection хорошая).
Рекомендация: Сценарий B при подтверждении доступа к RR. Если только Veepoo HRV — Сценарий A с дисклеймером и собственным L3-индексом (Stress Score Neiry), который сглаживает зависимость от их формулы.
Кирилл нашел методы, которые позволяют выгрузить эти накопленные массивы постфактум (исторически):
readRRIntervalByDay — Это наш Святой Грааль. Метод позволяет мобильному приложению подключиться к браслету и выкачать чистый массив всех R-R интервалов за выбранный день.
readHRVOrigin — Выгружает пакеты данных, внутри которых есть поле rrValue. Это альтернативный способ достать те же сырые данные.
convertRRData2RRLorenzInfo / getHrvAnalysisReport — Это внутренние функции SDK, которые Veepoo использует, чтобы на основе выгруженных RR-интервалов построить свои графики (Lorenz) и отчеты.
Дилемма 1 решена: Поскольку мы имеем доступ к массиву RR (пусть и историческому), мы можем официально выбрать Сценарий B — Собственный HRV на наших RR.
Собственная IP (Интеллектуальная собственность): Нам больше не нужно доверять закрытой китайской формуле (bod_hrv_char_t), состав которой неизвестен. Кирилл будет просто выкачивать массив RR через readRRIntervalByDay и отправлять его JSON-пакетом на бэкенд Арсению.
Уровень доверия (Маркер): Ваша метрика HRV официально переходит из статуса 🟡 «Условно» в статус 🟢 «Достоверно». Арсений сможет считать RMSSD, SDNN, LF/HF по любым международным медицинским стандартам на вашем сервере.
Единственное ограничение для дизайна
Из-за того, что данные выгружаются исторически (batch-sync), вы не сможете сделать экран, где график HRV или уровень стресса пересчитывается каждую секунду прямо на глазах у пользователя.
Эта механика идеально ложится на паттерны лидеров рынка (WHOOP, Oura, Polar Nightly Recharge). Пользователь спит в браслете, утром открывает приложение, данные R-R интервалов выгружаются за секунду, улетают на сервер, и сервер возвращает красивый Neiry Daily Score.
Скажите Кириллу, что это то, что нужно, и пусть он настраивает фоновую выгрузку этих массивов при каждом коннекте браслета с телефоном.
Дилемма 2. Дaвление — public-mode или private-mode с калибровкой?
SDK поддерживает два режима blood pressure:
DETECT_MODEL_PUBLIC — без калибровки, чисто алгоритм Veepoo. Юридический риск — измерение недостоверно (🟠).
DETECT_MODEL_PRIVATE — с пользовательской калибровкой по эталонной манжете в onboarding. Точность ±10–15 mmHg при свежей калибровке.
Рекомендация: Public-режим — за toggle «Экспериментально» с дисклеймером. Private-режим — основной путь для Care-сегмента (пенсионеры), с обязательной калибровкой манжетой в первой консультации с гериатром. Это даёт научно обоснованную метрику и оправдывает премиум-цену пакета Care.
Дилемма 3. Stress Score — Veepoo, свой, или оба?
Veepoo даёт свой Stress Score (часть OriginData). Алгоритм проприетарный, формула неизвестна. Большинство аналогов на рынке (Garmin, Fitbit) считают Stress на основе HRV-падения от baseline.
Рекомендация: Хранить и Veepoo, и собственный (через колонку source ENUM('veepoo','neiry_custom')). По умолчанию показывать Neiry-расчёт (он более прозрачен и контролируем), Veepoo — как референс / для совместимости с G-Band приложением.
Дилемма 4. Энергопотребление vs богатство данных
Каждое continuous-измерение (особенно SpO₂, BP) ускоряет разряд батареи в разы. Веепо рекомендует MODE1 PPG (15 мин/10 сек) как разумный компромисс.
Рекомендация: Профиль энергопотребления привязать к сегменту:
Corporate Office: PPG MODE1, SpO₂ только ночной, BP off — 25+ дней автономии (главный USP для массового сегмента).
Sport (VIGOR): PPG continuous во время тренировки, иначе MODE1. GPS только в режиме тренировки.
Care (пенсионеры): PPG MODE1, SpO₂ ночной + on-demand, BP в Private-mode 1–2 раза в день по расписанию.
Premium Recovery / биохакинг: PPG MODE2 (детальный), все метрики on, частая синхронизация. Автономии меньше — компенсируется ценностью данных.

9. Edge cases и failure modes
Скелетный раздел для итерации 2. Каждая метрика в Excel-реестре имеет колонку 'Edge cases', здесь — сводка общих паттернов.
Общие паттерны отказов
Браслет снят / неплотный контакт: wearing detection через PPG signal quality. Показывать last_known_value с timestamp, не пытаться экстраполировать.
Motion artifact (интенсивная тренировка, печатание): PPG HR может скакать на ±30 bpm. Решение — confidence score из SDK + фильтрация на backend.
BLE-разрыв связи: данные продолжают сохраняться на устройстве (внутренняя память), синхронизируются при следующем connection. Дашборд показывает «Последняя синхронизация: X минут назад».
Дневной сон, перелёты, сменная работа: Sleep detection ломается. Решение — manual override и user feedback loop для recalibration.
Низкий заряд (<15%): тяжёлые измерения (BP, continuous SpO₂) автоматически выключаются. UI уведомление пользователю.
