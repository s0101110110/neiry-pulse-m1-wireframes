## Блок A1. Wow-сценарий стенда M2 ✅

### Решение
**Двойной сценарий A + B4:**

1. **A — Тест Струпа** (~60 сек, планшет на стенде, B2B-нарратив «диспетчеры, охрана труда»). Гость может пройти сам.
2. **B4 — Беговая дорожка с офисным перформером в костюме** как постоянный hero. Перформер на дорожке весь день — корпоратная метафора «жизнь сотрудника под нагрузкой».

### Wow-механика «дельта»
Параллельные сессии: гость надевает свой браслет, видит, что у него метрики **не меняются** (зелёная зона), а у бегуна на дорожке — **ползут в красное**. Дельта между «спокойным корпоратом» и «бегуном» — это и есть Wow-момент за 5 секунд.

Дальше гость может:
- Пройти тест Струпа сам и увидеть свой шифт в жёлтую зону
- Получить персональный отчёт в Telegram через PIN-flow (уже спроектирован в `knowledge-base/milestones/m2.md`)

### Почему так
- **Корпоратный нарратив** попадает в P1 целевой аудитории Startup Village (инвесторы B2B). Костюм + дорожка = читается без объяснений.
- **Виральность:** инвесторы фотографируют и постят — органический PR.
- **Связь с метрикой:** ΔBPM от baseline в покое vs нагрузка — прямая иллюстрация Stress Trend / Burnout Risk из дашборда Корпорат.
- **Нулевая «постановка»:** гость в любой момент может надеть браслет и реально увидеть свои цифры рядом с бегуном (не подделанная демка).

### Логистика для PM
- Аренда беговой дорожки на 2 дня (~10-20 тыс ₽) — маркетинговая инвестиция.
- 2 актёра-перформера посменно (1 устанет за день) или паузы каждый час.
- Дорожку — в дальний угол стенда (шум), плазму — на основной площади.
- 2-3 браслета: 1 на перформере (постоянно), 2 на ротации для гостей с дезинфекцией.
- Планшет с приложением «Тест Струпа» — отдельная разработка/найти готовое.

### Открытые подвопросы
- ❓ Костюм перформера: классический пиджак или джинсы + рубашка? (визуальный код «офис»)
- ❓ 2 или 3 одновременных гостя — ТЗ Кирилла позволяет 2-3, нужно подтвердить с фронтом.

## Блок A2. Layout плазмы 42" ✅

### Решение
**L4 — Гибрид «шкала + карточки с sparkline»**, выбранный после визуального сравнения 4 вариантов в Visual Companion. Прототип: [`assets/2026-05-18-m2-plasma-layout-l4.html`](assets/2026-05-18-m2-plasma-layout-l4.html) — standalone, открывается без сервера.

### Структура плазмы
```
┌─ Neiry · Pulse Live ────────── 🔴 Live · Stress/Calmness Score ─┐
│                                                                  │
│              КОНЦЕНТРАЦИЯ · СТРЕСС                               │
│  ╔═══════════════════════════════════════════════════════════╗   │
│  ║ 👤22%      👨52%                              🏃78%        ║   │
│  ║ ===========================================================║   │
│  ║   зелёный    →   жёлтый   →    красный                    ║   │
│  ╚═══════════════════════════════════════════════════════════╝   │
│  Спокойствие 0%        Концентрация 50%       Острый стресс 100% │
│                                                                  │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐               │
│  │ 🏃 Бегун    │  │ 👨 Дмитрий  │  │ 👤 Алексей  │               │
│  │ КРАСНАЯ     │  │ ЖЁЛТАЯ      │  │ ЗЕЛЁНАЯ     │               │
│  │  168 BPM    │  │  98 BPM     │  │  76 BPM     │               │
│  │  ~~/\~~^~~  │  │  ~~~^~~~~~  │  │  ~~~~~~~~~  │ sparkline →  │
│  │ SpO₂ Δ И%   │  │ SpO₂ Δ И%   │  │ SpO₂ Δ И%   │               │
│  └─────────────┘  └─────────────┘  └─────────────┘               │
└──────────────────────────────────────────────────────────────────┘
```

### Модульная структура карточки (важно: системо-образующая)
Каждая карточка — 4 модуля. Можно включать/отключать без переверстки:
1. **Header** — имя (с устройства автоматически через auth C3) + статус-бейдж (зона)
2. **BPM-блок** — большая цифра + «+ΔN от baseline»
3. **Spark-chart** — sparkline пульса за последние ~60 сек, **ползёт вправо**, цвет линии = цвет зоны
4. **Meta-row** — `SpO₂` · `Δ Baseline` · `Индекс %`

Расширяемо: в будущем можно добавить HRV / температуру / Activity без переделки.

### Три состояния плазмы
1. **Только бегун** — одна карточка на весь экран, BPM-цифра 140pt. Работает всегда (даже если гостей нет, плазма не пустая).
2. **Бегун + 1 гость** — 50/50, главный Wow-кадр.
3. **Бегун + 2 гостя** — 33/33/33, демонстрирует «как разные люди реагируют по-разному».

Имена гостей берутся с устройства автоматически (`Pixel 7 · a3f2`) — решение auth C3 от 2026-05-18.

### Цветовая логика (медицинский светофор)
- 🟢 Зелёная зона (0-30%): спокойствие, парасимпатика, BPM ≈ baseline
- 🟡 Жёлтая зона (30-65%): когнитивная нагрузка, BPM +15-25 от baseline
- 🔴 Красная зона (65-100%): острый стресс / физическая нагрузка, BPM >+30 от baseline

Линейная шкала (а не зеркальная) — проще для гостя за 5 секунд.

### Почему L4 (а не L1/L2/L3)
- **L1 (дуэль 50/50 с цифрами)** — требует чтения двух цифр, не работает за 5 сек.
- **L2 (только шкала)** — теряет цифры BPM, инвесторам нужны и числа тоже.
- **L3 (только график)** — гость должен «читать» график, > 5 сек на понимание.
- **L4 (гибрид)** — шкала даёт мгновенное считывание, карточки дают доказательство цифрами. Лучшее из обоих.

### Системо-образующая ценность L4
PM явно отметил: layout становится частью **Admin + User Flow и питчинг-презентации**. Это не одноразовая кисочка для M2 — это **продуктовый язык Neiry**. Те же элементы (шкала-светофор, модульные карточки) можно переиспользовать:
- Mobile (главный экран в Corporate Mode)
- Neiry Unite дашборд (Team Readiness, Team Stress)
- Питч-слайды для инвесторов

## Блок A3.0. Архивный HRV на стенде M2 ✅

### Решение
**Показываем HRV только у якорей (Алексей, Илья), у гостей — заглушка «Нужны данные за ночь».**

На карточке плазмы — 5-й модуль (HRV ночной):
- **У якорей:** `RMSSD 56 мс · ✓ норма` (цвет по шкале 30-60 норма)
- **У гостей:** `ⓘ Нужны данные за ночь · 📲 Подключись к TG-боту`

Это «вариант 2 + вариант 4» из обсуждения: HRV не везде, но там, где есть — это «доказательство глубины» (мы не только live BPM, мы и про восстановление).

### Sync-цепочка для архивных RR (3 дня до выставки)

**Проблема:** дашборд ещё не написан, а данные за 2 дня уже копятся в браслетах якорей. Как загрузить в backend до 25.05?

**План V2 (минимум новой работы):**
1. **Браслеты на руках 24/7 (с сегодня)** — Алексей и Илья носят постоянно, **спят в них в ночь 24→25.05** (главное окно для RMSSD).
2. **Утром 25.05** — Алексей подключается через **существующий dev-UI** (тот, что уже работает — скриншот `sources/ui-references/HRV (chart).png`), вытягивает RR-массив за ночь, сохраняет JSON.
3. **Кирилл создаёт endpoint** `POST /admin/seed-rr` (один эндпоинт, 30 минут работы) — принимает JSON с RR-массивом и `user_id`, считает RMSSD на backend, кладёт в БД.
4. **Илья (iOS)** — если BLE-bridge на iOS ещё не готов, тянем его данные через **Android-телефон Алексея** (один раз). Илья надевает свой браслет на руку Алексея на 5 минут утром? Нет — проще: Илья просит Алексея забрать его браслет, синкнуть через dev-UI Android, и вернуть.
5. **Плазма читает готовый RMSSD из БД** — никакой live-расчёт не нужен.

### Что Кирилл должен сделать (3 пункта)
- `POST /admin/seed-rr {user_id, rr_intervals: [...], date}` — endpoint загрузки
- Расчёт RMSSD по Task Force 1996 (формула в `metrics/hrv.md`)
- Чтение готового RMSSD на плазме из той же таблицы, что и live BPM

### Что Илья и Алексей должны сделать
- ✅ Носить браслеты непрерывно с 22→25.05 (3 ночи)
- ✅ Не снимать на ночь — это критично для окна 00:00-07:00
- ✅ Утром 25.05 синк через dev-UI Android (раз для каждого)

### Открытые риски
- ⚠️ Что если у Ильи (iOS) браслет не отдаст данные через Android Алексея? — Fallback: показываем HRV только у Алексея, у Ильи как у гостя «Нужны данные».
- ⚠️ Что если RMSSD окажется в красной зоне (<15 мс) у якоря? Звоночек выгорания у нашего инженера за день до выставки — нужно подменить «бегуна» на здорового стороннего человека? — Решение: проверить RMSSD у обоих 24.05 утром, если у одного критика — оставить только второго на плазме.

### Открытый вопрос
- ❓ Показываем ли «вчерашний» RMSSD или скользящее среднее за 3 ночи? Рекомендация: одна цифра (вчерашняя) — гостю проще читать.

## Блок A4. Метрики и UI для Neiry Unite (HR-дашборд) ✅

### Решение
**Берём Никитин дизайн v4 как baseline целиком, без переделок.** Локальные правки — только в 3 точках.

### Иерархия решений (A4.1 → A4.5)

| Подвопрос | Решение | Комментарий |
|-----------|---------|-------------|
| **A4.1** Главная работа HR | **Operational Triage** | Утренний скан «кто сегодня в зоне риска», конкретные имена + actionable алерты |
| **A4.2** Формат сигнала | **B+C гибрид**: 4 композита в таблице + event-feed «Инсайты» | В прототипе уже разделены: таблица команды + feed внизу |
| **A4.3** Композиты в KPI strip | **Recovery / Stress / Sleep / Burnout** (4 cards) | Activity убираем (не wellness, → фитнес-вертикаль). Illness не в KPI — он event, его место в feed + per-row badge |
| **A4.3.1** Таймфреймы | **Today / 7d / 14d / 30d** | Period selector в правом верхнем |
| **A4.3.2** Default view | **Не нужно выбирать**: Никитин дизайн показывает и snapshot, и тренд одновременно | KPI = Today, charts = 7d, heatmap = и то, и другое |
| **A4.4** Event types в feed | **Sleep Deficit / Early Illness / Burnout Warning / Recovery Comeback** | 3 негативных + 1 позитивный |
| **A4.5** Drill-down | **Slide-in panel справа** (Никитин паттерн) | Не отдельная страница — контекст команды не теряется |

### Почему Никитин дизайн выиграл моё «split-screen hybrid» предложение

Я предлагал «верх = Today, низ = 7d sparkline». Никита сделал лучше: **один экран показывает 4 уровня детализации одновременно**:

1. **Top KPI strip** (5 → 4 после правок) = team snapshot прямо сейчас + Δ% trend
2. **Charts «Readiness и Stress за неделю»** = 7d trend линий
3. **Team heatmap** (10 цветных квадратов на сотрудника) = **гениальный паттерн**: в одной строке и snapshot (правый квадрат), и тренд (вся полоса)
4. **Инсайты feed** = event-based алерты с severity

Toggle 14d/30d прикручивается к period selector «12 апр – 18 апр» в правом верхнем.

### Drill-down (A4.5) — slide-in panel

Клик на сотрудника → выезжает справа панель:
- 5 donut-метрик крупно (с цветом по зонам)
- Стресс в течение дня (24h)
- Пульс за день (24h)
- HRV (RMSSD) за неделю
- 6 stat-карточек: RHR, HRV, Шаги, Sedentary, Active Min, Activity Score
- Health flags (Early Illness, Burnout Risk) с **рекомендацией Neiry-продуктов** (Headband + tVNS) — pattern сохраняем, контент уточняет PM (см. memory `project_neiry_product_bundle.md`)

### 3 локальные правки в Никитином дизайне

1. **KPI strip:** 5 → 4 cards. Activity убираем из team-level (не actionable для HR, идёт в фитнес-вертикаль)
2. **Illness Alert:** не в KPI strip, только в insights feed + per-row badge (Никита уже так сделал, фиксируем)
3. **Recommendations Neiry-продуктов в drill-down:** паттерн сохраняем, конкретный контент (Headband, tVNS) — отложен на M4+ (продукты ещё не готовы)

### Открытые вопросы (для будущих обсуждений)

- ❓ **Privacy / consent** — B2B HR-продукт показывает индивидуальную физиологию. Нужны opt-in флоу, политика данных, anonymization-режим для group-views? Юридический вопрос, отдельная сессия.
- ❓ **Что показываем на пустой команде** (новые клиенты, нет ещё данных) — onboarding UX?
- ❓ **Группы / Сравнение / Инсайты** (sidebar-tabs Никиты) — это отдельные экраны, не покрыты этим брейнштормом.
